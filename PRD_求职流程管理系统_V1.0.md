## 3. 项目概述

### 3.1 项目背景

对于计算机行业校招生和社招生而言，求职通常包含三个高度相关但长期割裂的动作：

1. 制作和持续修改简历。
2. 针对不同岗位进行投递。
3. 进入面试后跟进进度、复盘反馈并继续优化简历。

当前市场上多数工具只覆盖单点场景，例如简历编辑器、招聘网站、面试记录工具，但很少能把“简历版本 - 岗位投递 - 面试结果”三者串联起来。实际求职中，候选人往往会：

- 为不同岗位准备不同简历版本，但无法清楚知道某家公司投递时使用的是哪一版简历。
- 在收到面试反馈后，需要回溯到底是哪段经历、哪个项目表达方式产生了效果。
- 随着时间推进不断更新简历，但缺少版本记录与投递结果之间的关联分析。

因此，需要一个面向个人求职者的一体化求职流程管理系统，将简历制作、版本演进、岗位投递和面试跟进放在同一个产品中统一管理。

### 3.2 项目目标

本项目目标是打造一套个人求职工作台，帮助用户：

1. 更高效地管理不同求职方向下的简历仓库与版本。
2. 明确记录“某次投递对应的岗位、公司、渠道、使用简历版本”。
3. 追踪面试进度并沉淀面试反馈，形成可回溯的求职闭环。
4. 通过数据关联分析，帮助用户持续优化简历内容并提升求职效率。

### 3.3 产品定位

面向个人求职者的求职全流程管理工具，核心定位是“围绕简历版本的求职资产管理平台”。

### 3.4 目标用户

| 用户类型 | 特征 | 核心诉求 |
| --- | --- | --- |
| 校招生 | 同时投递多家公司、多岗位，简历需快速调整 | 快速生成多版简历，记录投递与面试进度 |
| 社招生 | 岗位方向明确，但会针对不同 JD 定制简历 | 管理多版本简历与投递效果，复盘反馈 |
| 高频求职者 | 同时推进多个流程，信息分散 | 统一视图查看简历、投递、面试全链路 |

### 3.5 核心价值

1. 提升求职组织效率，减少在文档、表格、聊天记录之间反复切换。
2. 形成“版本化”求职资产，避免简历更新后历史投递不可追溯。
3. 让投递结果和面试反馈反向驱动简历优化，提高求职质量。

## 4. 产品目标与衡量指标

### 4.1 业务目标

1. 完成个人求职流程的统一承载。
2. 先建立简历版本管理能力，再逐步扩展至投递与面试管理。
3. 通过简历版本与后续环节关联，形成求职分析基础数据。

### 4.2 北极星指标

- 用户成功维护并实际使用的简历版本数。

### 4.3 关键指标

| 指标 | 定义 |
| --- | --- |
| 简历仓库创建率 | 注册用户中创建至少 1 个简历仓库的比例 |
| 简历版本活跃率 | 创建至少 2 个版本并完成激活切换的用户比例 |
| 投递记录关联率 | 投递记录中绑定简历版本的占比 |
| 面试复盘完成率 | 进入面试的记录中填写面试反馈的占比 |

说明：当前版本已具备简历仓库和版本管理能力，后两项指标将在投递与面试模块上线后正式启用。

## 5. 版本范围

### 5.1 本期版本定义

本 PRD 对应的产品版本为 V1.0，采用“整体目标 + 分阶段落地”的方式：

1. V1.0 当前已完成范围：聚焦简历管理底座。
2. V1.1 计划范围：补齐岗位投递记录。
3. V1.2 计划范围：补齐面试管理与关联分析。

### 5.2 当前已完成范围

当前项目中已经完成的功能包括：

1. 用户注册、登录、个人信息维护。
2. 简历仓库管理，支持校招/社招/实习分类。
3. 简历版本管理，支持创建、编辑、查看、删除、激活。
4. Markdown 简历预览。
5. 简历 PDF 导出。
6. Mock API、前端路由、状态管理与基础页面搭建。

### 5.3 本期不包含

本版本暂不包含以下能力：

1. 岗位投递记录管理。
2. 面试流程跟进与复盘。
3. 投递结果与简历版本的效果分析。
4. AI 自动生成简历或自动改写功能。
5. 多端同步、消息提醒、日历集成。

## 6. 业务流程

### 6.1 完整目标流程

1. 用户维护个人基础信息。
2. 用户按求职方向创建简历仓库。
3. 用户在仓库下不断产出不同简历版本。
4. 用户选择某一激活版本用于具体岗位投递。
5. 用户记录投递信息并绑定简历版本。
6. 用户进入面试后记录轮次、时间、反馈和结果。
7. 用户根据投递与面试反馈继续优化简历版本。

### 6.2 当前版本实际支持流程

1. 注册/登录系统。
2. 维护个人信息和求职意向。
3. 创建简历仓库。
4. 在仓库下创建多个简历版本。
5. 编辑简历内容并实时预览。
6. 激活当前使用版本。
7. 导出 PDF 用于外部投递。

## 8. 功能详细说明

### 8.1 用户中心

#### 8.1.1 注册与登录

功能说明：

1. 支持用户注册账号并登录系统。
2. 登录后进入求职管理后台。

输入项：

- 用户名
- 密码

验收标准：

1. 用户可完成注册并正常登录。
2. 登录状态可用于访问简历相关页面。

#### 8.1.2 个人信息管理

功能说明：

1. 用户可维护姓名、年龄、邮箱、电话、求职意向、头像等信息。
2. 后续简历生成、投递记录和面试记录均可复用这些基础信息。

验收标准：

1. 用户可查看并更新个人信息。
2. 更新后信息可再次读取并保持一致。

### 8.2 简历仓库管理

#### 8.2.1 创建仓库

功能说明：

1. 用户可为不同求职方向创建独立的简历仓库。
2. 仓库需支持区分校招、社招与实习。
3. 仓库需包含目标岗位和仓库描述。

字段：

| 字段 | 说明 | 是否必填 |
| --- | --- | --- |
| 仓库名称 | 用于区分不同简历方向 | 是 |
| 求职类型 | 校招 / 社招 / 实习 | 是 |
| 目标岗位 | 如前端工程师、后端工程师 | 是 |
| 仓库描述 | 对仓库用途的补充说明 | 否 |

验收标准：

1. 用户可创建仓库并在列表中看到新增记录。
2. 列表支持搜索、筛选、分页。

#### 8.2.2 仓库列表与详情

功能说明：

1. 列表页展示全部简历仓库。
2. 可按关键词、求职类型筛选。
3. 详情页展示仓库基础信息及其下属简历版本列表。

验收标准：

1. 列表页能正确展示仓库数量与基础信息。
2. 详情页能展示版本数量、版本列表及仓库元数据。

### 8.3 简历版本管理

#### 8.3.1 创建与编辑版本

功能说明：

1. 一个简历仓库下允许存在多个版本。
2. 用户可输入版本标题、内容、备注。
3. 内容采用 Markdown 形式编辑，便于结构化表达项目经历与技能。

验收标准：

1. 用户可在指定仓库下创建新版本。
2. 用户可修改已有版本标题、内容、备注。
3. 保存后返回仓库详情页并能看到新版本。

#### 8.3.2 版本查看与激活

功能说明：

1. 用户可查看某个简历版本的完整内容。
2. 一个仓库同一时间仅允许一个版本为激活状态。
3. 激活版本代表当前默认用于外部投递的版本。

验收标准：

1. 版本详情页可展示标题、版本号、创建时间、备注和正文。
2. 激活一个版本后，原激活版本自动取消激活状态。

#### 8.3.3 版本追踪价值

业务要求：

1. 不同版本必须保留版本号和时间信息。
2. 后续投递记录必须能明确关联到具体简历版本，而不是只关联简历仓库。

说明：

当前版本已完成版本管理能力，并具备后续扩展投递关联的基础数据结构。

### 8.4 简历预览与导出

#### 8.4.1 Markdown 预览

功能说明：

1. 系统需将 Markdown 内容渲染为可阅读的简历排版。
2. 编辑页和查看页的内容表达应保持一致。

验收标准：

1. 标题、列表、段落、表格等基础 Markdown 语法可正常显示。
2. 用户无需手动转换格式即可预览简历效果。

#### 8.4.2 PDF 导出

功能说明：

1. 用户可将当前查看的简历版本导出为 PDF 文件，用于投递。
2. 文件名应包含简历标题与版本号，便于外部管理。

验收标准：

1. 点击导出后可下载 PDF 文件。
2. 导出内容不包含后台管理按钮等无关 UI。
3. 文件命名符合“标题_版本号.pdf”的规则。

### 8.5 投递管理（规划）

#### 8.5.1 功能目标

投递模块需解决“投了哪些岗位、用了哪一版简历、当前进展如何”的核心问题。

#### 8.5.2 计划字段

| 字段 | 说明 |
| --- | --- |
| 公司名称 | 投递公司 |
| 岗位名称 | 投递岗位 |
| 岗位类型 | 校招 / 社招 / 实习 |
| 投递渠道 | Boss、牛客、官网、内推等 |
| 投递日期 | 实际投递时间 |
| 简历版本 ID | 关联具体版本 |
| 投递状态 | 已投递、已查看、笔试、面试中、已拒绝、已录用 |
| 备注 | 记录特殊信息 |

#### 8.5.3 核心规则

1. 每条投递记录必须绑定一个简历版本。
2. 简历版本删除前，需校验是否已被投递记录引用。
3. 投递状态需支持时间维度的更新与回溯。

### 8.6 面试管理（规划）

#### 8.6.1 功能目标

面试模块需帮助用户跟进每次面试安排，并沉淀复盘信息，形成可用于后续简历优化的反馈库。

#### 8.6.2 计划字段

| 字段 | 说明 |
| --- | --- |
| 投递记录 ID | 面试关联的投递记录 |
| 面试轮次 | 一面、二面、HR 面、终面 |
| 面试时间 | 实际面试时间 |
| 面试形式 | 现场、视频、电话 |
| 面试结果 | 通过、待定、未通过 |
| 面试反馈 | 问题点、表现亮点、被追问内容 |
| 改进建议 | 需要补充到简历或表达中的内容 |

#### 8.6.3 核心规则

1. 一条投递记录可关联多轮面试。
2. 面试反馈应支持回溯到所使用的简历版本。
3. 用户可基于面试反馈继续迭代简历并形成新版本。

## 10. 数据流与系统设计

### 10.1 核心实体

当前已存在实体：

1. UserProfile
2. ResumeRepo
3. ResumeVersion

规划新增实体：

1. JobApplication
2. InterviewRecord

### 10.2 核心关联关系

```text
用户 1:N 简历仓库
简历仓库 1:N 简历版本
简历版本 1:N 投递记录（规划）
投递记录 1:N 面试记录（规划）
```

### 10.3 数据要求

1. 简历版本必须保留 `repoId`、`version`、`isActive`、创建时间和更新时间。
2. 投递记录上线后必须保留 `resumeVersionId`，不能只记录 `resumeRepoId`。
3. 面试记录上线后必须保留 `applicationId`，并可回溯对应的版本信息。

## 11. 异常场景与风险

### 11.1 异常场景

1. 用户删除仓库时，若仓库下存在版本，需要二次确认。
2. 用户导出 PDF 失败时，需给出明确失败提示。
3. 后续若版本已被投递记录引用，删除版本时需限制或要求迁移关联关系。
4. 后续若投递状态频繁变化，需保留状态更新时间，避免信息失真。

### 11.2 风险点

1. 当前版本尚未接入真实后端，数据依赖 Mock，无法满足长期持久化需求。
2. 目前只完成简历管理，尚未真正形成“投递 - 面试 - 分析”的闭环。
3. 如果后续数据模型未提前考虑版本关联，投递与面试分析会出现历史数据断裂。

### 11.3 风险应对

1. 后端建模时优先保证版本、投递、面试三者的主外键关系。
2. 投递模块上线时同步设计状态流转和操作日志。
3. 面试模块上线时加入复盘模板，提升结构化记录质量。

## 12. 测试验收与上线建议

### 12.1 当前版本验收范围

1. 用户可注册、登录并维护个人信息。
2. 用户可创建、编辑、删除简历仓库。
3. 用户可在仓库下创建、编辑、查看、删除简历版本。
4. 用户可激活指定版本，且同仓库仅一个激活版本生效。
5. 用户可预览 Markdown 简历内容。
6. 用户可导出 PDF，且文件名符合规则。

### 12.2 后续版本验收重点

1. 每条投递记录必须关联一个明确的简历版本。
2. 同一岗位在不同时间再次投递时，应可区分使用版本与投递结果。
3. 面试记录应能完整回溯到投递记录和简历版本。
4. 分析页面应能统计不同版本的投递转化和面试转化。

### 12.3 上线建议

1. 先以 V1.0 形式上线“简历管理系统”能力，验证用户是否愿意持续维护简历版本。
2. 在获得真实使用反馈后，再推进投递和面试模块，以保证产品演进节奏清晰。
3. 对外描述上可定位为“求职流程管理系统”，对内研发阶段按“简历管理底座 -> 投递 -> 面试 -> 分析”推进。
