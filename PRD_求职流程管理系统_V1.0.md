## 3. 项目概述

### 3.1 项目背景

对于计算机行业校招生和社招生而言，求职通常包含三个高度相关但长期割裂的动作：

1. 制作和持续修改简历。
2. 针对不同岗位进行投递。
3. 进入面试后跟进进度、复盘反馈并继续优化简历。

当前市场上多数工具只覆盖单点场景，例如简历编辑器、招聘网站、面试记录工具，但很少能把“简历版本 - 岗位投递 - 面试结果”三者串联起来。实际求职中，候选人往往会：

- 为不同岗位准备不同简历版本，但无法清楚知道某家公司投递时使用的是哪一版简历。
- 在收到面试反馈后，需要回溯到底是哪段经历、哪个项目表达方式产生了效果。
- 随着时间推进不断更新简历，但缺少版本记录与投递结果之间的关联分析。

因此，需要一个面向个人求职者的一体化求职流程管理系统，将简历制作、版本演进、岗位投递和面试跟进放在同一个产品中统一管理。

### 3.2 项目目标

本项目目标是打造一套个人求职工作台，帮助用户：

1. 更高效地管理不同求职方向下的简历仓库与版本。
2. 明确记录“某次投递对应的岗位、公司、渠道、使用简历版本”。
3. 追踪面试进度并沉淀面试反馈，形成可回溯的求职闭环。
4. 通过数据关联分析，帮助用户持续优化简历内容并提升求职效率。

### 3.3 产品定位

面向个人求职者的求职全流程管理工具，核心定位是“围绕简历版本的求职资产管理平台”。

### 3.4 目标用户

| 用户类型 | 特征 | 核心诉求 |
| --- | --- | --- |
| 校招生 | 同时投递多家公司、多岗位，简历需快速调整 | 快速生成多版简历，记录投递与面试进度 |
| 社招生 | 岗位方向明确，但会针对不同 JD 定制简历 | 管理多版本简历与投递效果，复盘反馈 |
| 实习求职者 | 需要持续收集实习机会并快速调整简历 | 区分实习简历仓库，沉淀岗位与投递进展 |
| 高频求职者 | 同时推进多个流程，信息分散 | 统一视图查看简历、投递、面试全链路 |

### 3.5 核心价值

1. 提升求职组织效率，减少在文档、表格、聊天记录之间反复切换。
2. 形成“版本化”求职资产，避免简历更新后历史投递不可追溯。
3. 让投递结果和面试反馈反向驱动简历优化，提高求职质量。

## 4. 产品目标与衡量指标

### 4.1 业务目标

1. 完成个人求职流程的统一承载。
2. 先建立简历版本管理能力，再逐步扩展至投递与面试管理。
3. 通过简历版本与后续环节关联，形成求职分析基础数据。

### 4.2 北极星指标

- 用户成功维护并实际使用的简历版本数。

### 4.3 关键指标

| 指标 | 定义 |
| --- | --- |
| 简历仓库创建率 | 注册用户中创建至少 1 个简历仓库的比例 |
| 简历版本活跃率 | 创建至少 2 个版本并完成激活切换的用户比例 |
| 投递记录关联率 | 投递记录中绑定简历版本的占比 |
| 面试复盘完成率 | 进入面试的记录中填写面试反馈的占比 |

说明：当前版本已具备投递与面试记录能力；投递记录关联率和面试复盘完成率可基于现有 Mock 数据验证，后续在真实后端与分析看板上线后正式统计。

## 5. 版本范围

### 5.1 本期版本定义

本 PRD 对应的产品版本为 V1.0，采用“整体目标 + 分阶段落地”的方式：

1. V1.0 已完成范围：简历管理底座。
2. V1.1 已完成范围：岗位库、投递记录、面试进展、个人信息预填充与注册体验完善。
3. 后续计划范围：真实后端持久化、分析看板、提醒与 AI 辅助能力。

### 5.2 当前已完成范围

当前项目中已经完成的功能包括：

1. 用户注册、登录、邮箱验证码注册、企业级登录注册页和登录页注册入口。
2. 账号级个人信息维护，支持顶层入口、结构化表单、教育经历、账号隔离 Mock 和创建简历版本时自动生成默认草稿。
3. 简历仓库管理，支持校招 / 社招 / 实习分类。
4. 简历版本管理，支持创建、编辑、查看、删除、激活、Markdown 预览和 PDF 导出。
5. 岗位库管理，支持岗位列表、详情、手动录入、链接解析导入、编辑、删除、查看原岗位。
6. 岗位与投递联动，支持从岗位创建投递记录、投递创建时选择岗位并保留岗位快照。
7. 投递记录管理，支持基于简历版本创建投递记录、查看详情、编辑、删除和版本关联展示。
8. 面试进展管理，支持投递记录下的面试阶段历史、流程节点展示、状态同步、面试小记、总结与评分。
9. Mock API、前端路由、状态管理与基础页面搭建。

### 5.3 本期不包含

本版本暂不包含以下能力：

1. 真实后端数据库和长期持久化。
2. 真实邮箱验证码发送服务，当前仍为 Mock-first。
3. 真实岗位网页抓取和解析服务，当前前端只消费 Mock 解析 JSON。
4. 投递结果与简历版本的统计分析看板。
5. AI 自动生成简历、自动改写或岗位匹配能力。
6. 多端同步、消息提醒、日历集成。

## 6. 业务流程

### 6.1 完整目标流程

1. 用户维护个人基础信息。
2. 用户按求职方向创建简历仓库。
3. 用户在仓库下不断产出不同简历版本。
4. 用户选择某一激活版本用于具体岗位投递。
5. 用户记录投递信息并绑定简历版本。
6. 用户进入面试后记录轮次、时间、反馈和结果。
7. 用户根据投递与面试反馈继续优化简历版本。

### 6.2 当前版本实际支持流程

1. 用户通过登录页进入系统；未注册用户可从登录页进入注册页。
2. 用户使用用户名、密码、邮箱和邮箱验证码完成注册，注册邮箱写入账号个人信息。
3. 用户维护个人信息、求职意向、头像、所在地、个人优势和教育经历。
4. 用户按求职方向创建简历仓库，可选择校招、社招或实习类型。
5. 用户在仓库下创建多个简历版本；新建版本时可基于个人信息自动生成默认简历草稿。
6. 用户编辑 Markdown 简历内容，切换编辑 / 预览模式，查看版本详情并导出 PDF。
7. 用户维护岗位库，可手动录入岗位或通过来源链接解析导入岗位信息。
8. 用户从岗位详情或简历版本详情创建投递记录，系统自动带入简历版本或岗位信息。
9. 用户在投递记录详情中维护面试进展、流程节点状态、面试小记、总结和评分。

## 8. 功能详细说明

### 8.1 用户中心

#### 8.1.1 注册与登录

功能说明：

1. 支持用户通过用户名、密码、邮箱和邮箱验证码注册账号。
2. 注册邮箱为必填字段，验证码用于前端注册页交互校验，当前阶段通过 Mock-first 实现。
3. 注册成功后，邮箱写入账号个人资料，后续可在个人信息页展示。
4. 登录页提供清晰的注册入口，未注册用户可跳转到注册页。
5. 登录页和注册页采用统一企业级认证页布局，包含品牌说明区和表单卡片。
6. 登录后进入求职管理后台。

输入项：

- 用户名
- 密码
- 邮箱
- 邮箱验证码

验收标准：

1. 用户可发送邮箱验证码并完成注册。
2. 邮箱格式、验证码必填和密码确认校验可用。
3. 用户注册后可正常登录。
4. 登录状态可用于访问简历、岗位、投递和个人信息相关页面。
5. 登录页可见注册入口，点击后进入注册页。
6. 登录页不展示默认演示账号密码文案，用户名、密码与显示 / 隐藏密码控件对齐且间距合理。

#### 8.1.2 个人信息管理

功能说明：

1. 用户可通过表单维护姓名、年龄、邮箱、电话、求职意向、头像、所在地、个人优势等信息。
2. 用户可维护多段教育经历，每段包含学校、学历、专业、入学时间和毕业时间；同一种学历只允许添加一段经历。
3. 个人信息页作为一级入口展示，不挂在“简历管理”下。
4. 创建简历版本时，系统可基于这些资料自动生成默认简历内容草稿，用户仍可继续编辑。
5. 投递记录和面试记录可通过账号、简历版本和投递链路间接复用这些基础信息。

验收标准：

1. 用户可查看并更新个人信息。
2. 更新后信息可再次读取并保持一致。
3. 不同 Mock 账号读取和更新的个人资料互不影响。
4. 教育经历支持添加、删除、保存，并拦截重复学历和不合法时间范围。
5. 创建简历版本时可自动生成默认草稿，且编辑已有版本不会被覆盖。

### 8.2 简历仓库管理

#### 8.2.1 创建仓库

功能说明：

1. 用户可为不同求职方向创建独立的简历仓库。
2. 仓库需支持区分校招、社招与实习。
3. 仓库需包含目标岗位和仓库描述。

字段：

| 字段 | 说明 | 是否必填 |
| --- | --- | --- |
| 仓库名称 | 用于区分不同简历方向 | 是 |
| 求职类型 | 校招 / 社招 / 实习 | 是 |
| 目标岗位 | 如前端工程师、后端工程师 | 是 |
| 仓库描述 | 对仓库用途的补充说明 | 否 |

验收标准：

1. 用户可创建仓库并在列表中看到新增记录。
2. 列表支持搜索、筛选、分页。

#### 8.2.2 仓库列表与详情

功能说明：

1. 列表页展示全部简历仓库。
2. 可按关键词、求职类型筛选。
3. 详情页展示仓库基础信息及其下属简历版本列表。

验收标准：

1. 列表页能正确展示仓库数量与基础信息。
2. 详情页能展示版本数量、版本列表及仓库元数据。

### 8.3 简历版本管理

#### 8.3.1 创建与编辑版本

功能说明：

1. 一个简历仓库下允许存在多个版本。
2. 用户可输入版本标题、内容、备注。
3. 内容采用 Markdown 形式编辑，便于结构化表达项目经历与技能。

验收标准：

1. 用户可在指定仓库下创建新版本。
2. 用户可修改已有版本标题、内容、备注。
3. 保存后返回仓库详情页并能看到新版本。

#### 8.3.2 版本查看与激活

功能说明：

1. 用户可查看某个简历版本的完整内容。
2. 一个仓库同一时间仅允许一个版本为激活状态。
3. 激活版本代表当前默认用于外部投递的版本。

验收标准：

1. 版本详情页可展示标题、版本号、创建时间、备注和正文。
2. 激活一个版本后，原激活版本自动取消激活状态。

#### 8.3.3 版本追踪价值

业务要求：

1. 不同版本必须保留版本号和时间信息。
2. 投递记录必须能明确关联到具体简历版本，而不是只关联简历仓库。

说明：

当前版本已完成版本管理能力，并已接入投递记录关联展示和从简历版本创建投递记录的流程。

### 8.4 简历预览与导出

#### 8.4.1 Markdown 预览

功能说明：

1. 系统需将 Markdown 内容渲染为可阅读的简历排版。
2. 编辑页和查看页的内容表达应保持一致。

验收标准：

1. 标题、列表、段落、表格等基础 Markdown 语法可正常显示。
2. 用户无需手动转换格式即可预览简历效果。

#### 8.4.2 PDF 导出

功能说明：

1. 用户可将当前查看的简历版本导出为 PDF 文件，用于投递。
2. 文件名应包含简历标题与版本号，便于外部管理。

验收标准：

1. 点击导出后可下载 PDF 文件。
2. 导出内容不包含后台管理按钮等无关 UI。
3. 文件命名符合“标题_版本号.pdf”的规则。

### 8.5 岗位库管理

#### 8.5.1 岗位列表与详情

功能说明：

1. 岗位库作为独立一级模块展示，不挂在“简历管理”下。
2. 用户可查看岗位卡片列表，并按关键词、来源平台、岗位状态筛选。
3. 岗位卡片展示公司名称、岗位名称、base、薪资范围、来源平台、岗位状态和创建时间。
4. 岗位详情页展示岗位摘要、基础信息、操作按钮和关联投递记录。
5. 用户可查看、编辑、删除岗位，也可打开来源链接查看原岗位。

验收标准：

1. 岗位列表页可正常加载、筛选和分页。
2. 岗位卡片布局与简历仓库卡片风格保持一致。
3. 岗位详情页可查看岗位基础信息和关联投递记录。
4. “查看原岗位”可打开 `sourceUrl`。

#### 8.5.2 岗位创建、编辑与链接解析

功能说明：

1. 用户可手动录入岗位信息。
2. 用户可输入岗位来源链接并调用解析接口，由 Mock 返回结构化岗位 JSON。
3. 解析成功后，系统将岗位字段回填到表单，用户可继续修改。
4. 前端只消费解析接口返回结果，不负责真实网页抓取和解析。

字段：

| 字段 | 说明 |
| --- | --- |
| 公司名称 | 岗位所属公司 |
| 公司介绍 | 公司补充信息 |
| 岗位名称 | 招聘岗位 |
| 岗位要求 | JD 或要求描述 |
| base | 工作地点 |
| 薪资范围 | 岗位薪资 |
| 来源平台 | Boss / 官网 / 牛客 / 内推 / 其他 |
| 来源链接 | 原始岗位 URL |
| 岗位状态 | 待投递 / 已投递 / 不合适 / 已关闭 |
| 备注 | 用户补充信息 |

验收标准：

1. 手动创建和编辑岗位可用。
2. 链接解析导入可回填岗位字段。
3. 解析后字段仍可手动编辑。

### 8.6 投递管理

#### 8.6.1 功能目标

投递模块解决“投了哪些岗位、用了哪一版简历、当前进展如何”的核心问题，已接入简历版本详情页和岗位库详情页。

#### 8.6.2 核心字段

| 字段 | 说明 |
| --- | --- |
| 公司名称 | 投递公司 |
| 岗位名称 | 投递岗位 |
| 公司介绍 | 公司补充信息 |
| 岗位要求 | JD 或要求描述 |
| base | 工作地点 |
| 薪资范围 | 岗位薪资 |
| 投递渠道 | Boss、牛客、官网、内推等 |
| 投递日期 | 实际投递时间 |
| 简历版本 ID | 关联具体版本 |
| 岗位 ID | 可选，关联岗位库岗位 |
| 岗位快照 | 保存投递当时的岗位字段 |
| 投递状态 | 已投递、已查看、笔试、面试中、已拒绝、已录用 |
| 面试小记 | 面试过程记录 |
| 面试总结 | 投递复盘总结 |
| 个人评分 | 简历匹配度、面试表现、岗位意向度、综合评价 |
| 备注 | 记录特殊信息 |

#### 8.6.3 核心规则

1. 每条投递记录必须绑定一个简历版本。
2. 投递记录可关联岗位库岗位，并保留岗位快照，避免岗位库后续修改影响历史投递语义。
3. 简历版本删除前，需校验是否已被投递记录引用。
4. 投递状态需与面试进展保持同步。
5. 用户可从简历版本详情页或岗位详情页进入投递创建流程。

验收标准：

1. 简历版本详情页可展示关联投递记录。
2. 从简历版本详情页创建投递时自动带入当前简历版本。
3. 从岗位详情页创建投递时自动带入岗位 ID 和岗位字段。
4. 投递记录详情页可查看、编辑投递信息、面试小记、总结和评分。
5. 已有关联投递记录的简历版本不可静默删除。

### 8.7 面试管理

#### 8.7.1 功能目标

面试模块帮助用户跟进每次投递的面试进展，并沉淀复盘信息，形成可用于后续简历优化的反馈库。

#### 8.7.2 核心字段

| 字段 | 说明 |
| --- | --- |
| 投递记录 ID | 面试关联的投递记录 |
| 阶段 | 简历投递、简历评估、笔试、一面、二面、终面、HR 面、Offer、拒绝等 |
| 发生时间 | 阶段发生或记录时间 |
| 结果 | 通过、待定、未通过 |
| 面试官或团队 | 可选，记录面试方信息 |
| 备注 | 问题点、表现亮点、被追问内容、改进建议 |

#### 8.7.3 核心规则

1. 一条投递记录可关联多轮面试。
2. 新增或更新面试进展时，需要同步投递记录当前状态。
3. 投递详情页以流程节点展示阶段状态：
   - 已完成节点：绿色
   - 当前节点：绿色高亮
   - 失败节点：红色
   - 未开始节点：灰色
4. 面试反馈应支持回溯到投递记录和所使用的简历版本。
5. 用户可基于面试反馈继续迭代简历并形成新版本。

验收标准：

1. 投递记录详情页可新增、编辑、删除面试进展。
2. 当前状态与面试进展同步。
3. 面试流程节点颜色和高亮规则正确。
4. 面试小记、总结和评分可保存并回显。

## 10. 数据流与系统设计

### 10.1 核心实体

当前已存在实体：

1. UserProfile
2. ResumeRepo
3. ResumeVersion
4. JobPosting
5. ResumeApplication
6. InterviewProgress

### 10.2 核心关联关系

```text
用户 1:N 简历仓库
简历仓库 1:N 简历版本
用户 1:1 个人信息
简历版本 1:N 投递记录
岗位库岗位 1:N 投递记录
投递记录 1:N 面试进展
```

### 10.3 数据要求

1. 简历版本必须保留 `repoId`、`version`、`isActive`、创建时间和更新时间。
2. 投递记录必须保留 `resumeVersionId`，不能只记录 `resumeRepoId`。
3. 投递记录关联岗位库时必须同时保留 `jobPostingId` 和岗位快照字段。
4. 面试进展必须保留 `applicationId`，并可回溯对应的投递记录和简历版本信息。
5. Mock 个人信息必须按当前 token 对应账号隔离读取和更新。

## 11. 异常场景与风险

### 11.1 异常场景

1. 用户删除仓库时，若仓库下存在版本，需要二次确认。
2. 用户导出 PDF 失败时，需给出明确失败提示。
3. 若版本已被投递记录引用，删除版本时需限制或要求迁移关联关系。
4. 若投递状态频繁变化，需通过面试进展保留状态更新时间，避免信息失真。
5. 岗位来源链接为空或失效时，查看原岗位需给出合理提示或不展示操作。
6. 邮箱验证码发送失败或验证码错误时，注册页需明确提示。

### 11.2 风险点

1. 当前版本尚未接入真实后端，数据依赖 Mock，无法满足长期持久化需求。
2. 邮箱验证码和岗位链接解析仍为 Mock-first，不能代表真实生产能力。
3. 目前已形成“简历 - 岗位 - 投递 - 面试”的前端闭环，但尚未形成统计分析闭环。
4. 如果后续真实后端模型未保持版本、投递、岗位快照和面试进展关联，分析会出现历史数据断裂。

### 11.3 风险应对

1. 后端建模时优先保证版本、岗位、投递、面试进展之间的主外键关系。
2. 真实邮件与岗位解析能力上线时，保持前端 API 响应结构兼容。
3. 后续分析模块应基于投递记录、岗位快照和面试进展做统计，不直接依赖可变岗位库字段。

## 12. 测试验收与上线建议

### 12.1 当前版本验收范围

1. 用户可注册、登录并维护个人信息。
2. 用户可创建、编辑、删除简历仓库。
3. 用户可在仓库下创建、编辑、查看、删除简历版本，创建时可自动带入个人信息草稿。
4. 用户可激活指定版本，且同仓库仅一个激活版本生效。
5. 用户可预览 Markdown 简历内容。
6. 用户可导出 PDF，且文件名符合规则。
7. 用户可维护岗位库，并通过手动录入或链接解析导入岗位。
8. 用户可从岗位或简历版本创建投递记录，并保留简历版本关联、岗位关联和岗位快照。
9. 用户可在投递记录中维护面试进展、当前状态、面试小记、总结和评分。
10. 登录页可进入注册页，注册页支持邮箱验证码注册。

### 12.2 后续版本验收重点

1. 真实后端接入后，每条投递记录必须继续关联一个明确的简历版本。
2. 同一岗位在不同时间再次投递时，应可区分使用版本与投递结果。
3. 面试记录应能完整回溯到投递记录和简历版本。
4. 分析页面应能统计不同版本的投递转化和面试转化。
5. 邮箱验证码和岗位解析从 Mock 切换到真实服务时，前端体验不应退化。

### 12.3 上线建议

1. 当前前端已具备“简历管理 + 岗位库 + 投递记录 + 面试进展”的 Mock 闭环，可作为内部演示和流程验证版本。
2. 对外上线前优先补齐真实后端持久化、真实邮箱验证码、岗位解析服务和基础权限策略。
3. 后续产品演进按“真实后端化 -> 数据分析 -> 提醒与 AI 辅助”推进。

## 14. 待确认事项

1. 产品对外名称最终是继续使用“简历管理系统”，还是升级为“求职流程管理系统”。
2. 投递记录是否需要支持同一岗位多次投递的专门视图。
3. 面试模块是否需要接入日历提醒或通知能力。
4. 分析模块优先输出版本维度分析，还是公司 / 岗位维度分析。
5. 真实邮箱服务和岗位解析服务的后端方案、风控策略和错误码规范。
