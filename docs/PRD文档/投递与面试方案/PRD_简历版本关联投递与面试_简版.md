# 简历版本关联投递与面试功能 PRD

## 1. 基本信息

| 项目名称 | 文档版本 | 创建日期 | 适用平台 |
| --- | --- | --- | --- |
| 简历版本关联投递与面试功能 | V0.1 | 2026-04-26 | Web 后台 |

## 2. 需求背景

当前系统已经具备简历仓库、简历版本、Markdown 预览和 PDF 导出能力，但用户完成简历制作后，后续的岗位投递和面试记录仍然缺少统一管理。

实际求职过程中，一个简历版本往往会被投递到多个岗位；同一个岗位也会产生多轮面试进展、面试小记和个人复盘评价。如果这些信息无法和具体简历版本关联，用户就很难回答以下问题：

1. 某个简历版本实际投递了哪些公司和岗位。
2. 哪一版简历带来了更多面试机会。
3. 某个岗位的面试反馈是否说明简历内容需要继续调整。
4. 自己在不同岗位面试中的表现和问题点分别是什么。

因此需要新增“投递记录管理”能力，并与“简历版本”建立清晰关联，形成简历版本到岗位投递、面试进展、复盘总结的完整链路。

## 3. 需求目标

1. 支持每个简历版本关联多个投递记录。
2. 支持每条投递记录记录岗位信息、公司信息和薪资 base 等关键字段。
3. 支持围绕投递记录持续维护面试进展、面试小记和总结。
4. 支持用户为每次投递或面试进行主观评价打分，方便后续复盘。

## 4. 适用用户

1. 校招生：需要同时投递多家公司并跟踪进展。
2. 社招生：需要针对不同岗位记录反馈并优化简历版本。
3. 高频求职者：希望把简历、投递、面试统一管理。

## 5. 核心功能范围

### 5.1 简历版本关联投递记录

功能说明：

1. 一个简历版本可以关联多条投递记录。
2. 每条投递记录必须明确归属到某一个简历版本。
3. 用户可以从简历版本详情页查看该版本下的全部投递记录。

核心价值：

1. 明确知道某版简历的实际使用范围。
2. 便于后续统计该版本带来的投递和面试效果。

### 5.2 投递记录管理

每条投递记录需至少包含以下信息：

| 字段 | 说明 | 是否必填 |
| --- | --- | --- |
| 简历版本 ID | 关联的简历版本 | 是 |
| 公司名称 | 投递公司名称 | 是 |
| 公司介绍 | 公司基础介绍、业务方向、规模等 | 否 |
| 岗位名称 | 如前端工程师、后端工程师 | 是 |
| 岗位要求 | JD 摘要、核心要求、加分项 | 否 |
| base | 工作城市或工作地点 | 否 |
| 薪资范围 | 如 20k-30k x 14 | 否 |
| 投递渠道 | Boss、官网、内推、牛客、猎头等 | 否 |
| 投递日期 | 实际投递日期 | 是 |
| 当前状态 | 已投递、已查看、笔试、面试中、已拒绝、已录用等 | 是 |
| 备注 | 其他补充信息 | 否 |

支持操作：

1. 新增投递记录。
2. 编辑投递记录。
3. 删除投递记录。
4. 在列表中查看投递状态。

### 5.3 面试进展管理

功能说明：

1. 每条投递记录下可以持续更新面试进展。
2. 面试进展支持按时间顺序记录多个阶段。
3. 用户可以记录每一轮面试的结果和关键信息。

建议支持的进展状态：

1. 已投递
2. 已查看
3. 笔试
4. 一面
5. 二面
6. 三面 / 终面
7. HR 面
8. Offer
9. 已拒绝
10. 已结束

每次面试进展建议包含字段：

| 字段 | 说明 |
| --- | --- |
| 进展阶段 | 如一面、二面、HR 面 |
| 时间 | 实际发生时间 |
| 结果 | 通过、待定、未通过 |
| 面试官/部门 | 可选记录 |
| 备注 | 过程补充说明 |

### 5.4 面试小记与总结

功能说明：

1. 用户可针对每条投递记录补充面试小记。
2. 面试小记用于记录题目、追问点、回答表现、失误点和亮点。
3. 支持在投递详情中持续累积和更新。

建议内容结构：

1. 本轮面试问了什么。
2. 哪些问题回答得好。
3. 哪些问题回答得不好。
4. 面试暴露出的简历问题或表达问题。
5. 后续应补充的知识点或项目表达。

### 5.5 个人面试评价打分

功能说明：

1. 用户可对每条投递或每轮面试做主观评价。
2. 打分用于帮助用户复盘自己的准备程度和临场发挥。

建议评分方式：

| 维度 | 说明 | 分值建议 |
| --- | --- | --- |
| 简历匹配度 | 简历与岗位要求是否匹配 | 1-5 分 |
| 面试表现 | 回答流畅度、逻辑性、表达效果 | 1-5 分 |
| 岗位意向度 | 用户对该岗位的主观兴趣 | 1-5 分 |
| 综合评价 | 整体主观评分 | 1-10 分 |

补充说明：

1. 评分可以先采用手动录入，不做自动计算。
2. 后续可基于评分和状态数据做统计分析。

## 6. 核心业务流程

### 6.1 主流程

1. 用户进入某个简历版本详情页。
2. 用户为该版本新增一个或多个投递记录。
3. 用户填写公司信息、岗位要求、base、投递渠道等信息。
4. 当投递进入下一阶段时，更新面试进展。
5. 每轮面试结束后记录面试小记和个人评价打分。
6. 用户根据面试反馈继续修改简历，形成新版本。

### 6.2 关键关联关系

```text
简历仓库 1:N 简历版本
简历版本 1:N 投递记录
投递记录 1:N 面试进展记录
投递记录 1:1 或 1:N 面试总结/评价
```

## 7. 页面建议

### 7.1 简历版本详情页新增内容

建议在当前简历版本详情页增加“关联投递记录”模块，展示：

1. 投递记录总数。
2. 每条投递记录的公司、岗位、base、当前状态。
3. 新增投递记录按钮。
4. 查看详情、编辑、删除入口。

### 7.2 投递记录详情页

建议包含以下区域：

1. 基本信息区：公司名称、岗位名称、公司介绍、岗位要求、base、薪资范围、投递渠道、投递时间。
2. 面试进展区：时间线形式展示各轮进度。
3. 面试小记区：记录问题、总结、复盘。
4. 个人评价区：展示评分和主观结论。

## 8. 核心规则

1. 投递记录必须关联到一个明确的简历版本，不能脱离版本独立存在。
2. 一个简历版本允许关联多个投递记录。
3. 删除简历版本前，如果已经存在投递记录，需要先提示用户并限制直接删除。
4. 投递状态和面试进展应支持持续更新，保留最新状态。
5. 面试小记和评分允许用户后补录入和多次编辑。

## 9. 非功能要求

1. 列表和详情页应支持基础的加载态和空状态。
2. 投递记录和面试记录字段应便于后续扩展统计分析。
3. 文案应尽量简洁，适合个人用户快速录入。

## 10. 验收标准

1. 用户可在某个简历版本下成功新增多条投递记录。
2. 每条投递记录可填写公司介绍、岗位要求、base 等信息。
3. 用户可更新投递状态和新增面试进展。
4. 用户可填写面试小记总结。
5. 用户可录入个人面试评价和打分。
6. 用户可在简历版本维度查看该版本关联的全部投递记录。

## 11. 后续可拓展方向

1. 按简历版本统计投递数、面试数、Offer 数。
2. 分析不同简历版本的面试转化率。
3. 增加岗位收藏、提醒、日历同步功能。
4. 增加 AI 复盘总结和简历优化建议。
