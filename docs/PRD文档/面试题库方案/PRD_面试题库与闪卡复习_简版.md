# 面试题库与闪卡复习功能 PRD

## 1. 基本信息

| 项目名称 | 文档版本 | 创建日期 | 适用平台 |
| --- | --- | --- | --- |
| 面试题库与闪卡复习功能 | V0.1 | 2026-05-09 | Web 后台 |

## 2. 需求背景

当前系统已经具备简历仓库、岗位库、投递记录和面试进展管理能力，用户可以记录一次求职从简历准备、岗位收藏、投递到面试推进的完整过程。

但在实际面试复盘中，“被问到什么题”“哪些题答得不好”“哪些知识点高频出现”“面试前应该优先复习什么”仍然主要依赖用户在投递记录中的面试小记自由输入。自由文本虽然灵活，但存在几个问题：

1. 面试题无法作为独立知识资产沉淀。
2. 同一道题在多场面试中出现时，难以统一统计频率。
3. 用户很难按标签、难度、题型、掌握状态筛选复习内容。
4. 面试前复习缺少轻量化、可重复使用的闪卡入口。
5. 面试小记中的题目和复习资料没有形成闭环。

因此需要新增“面试题库”模块，将面试题从面试小记中抽象为可管理、可关联、可复习、可统计的独立对象，并与投递记录和面试进展形成连接。

## 3. 需求目标

1. 新增独立的面试题库模块，用于统一管理面试题、答案解析和复习状态。
2. 支持题目按标签、难度、题型、掌握状态进行分类和筛选。
3. 支持将题目关联到具体投递记录或某一轮面试进展，用于统计真实面试出现频率。
4. 支持随机抽题和条件抽题的闪卡复习流程，帮助用户面试前快速训练。
5. 支持用户在闪卡复习后标记掌握情况，形成后续复习优先级。
6. 支持题目收藏，方便用户将重点题加入优先复习范围。
7. 保持与现有岗位库、投递记录、面试进展模块的风格和数据关系一致。

## 4. 方案定位

本功能定位为“真实面试反馈驱动的个人题库”，不是通用八股题收藏夹。

核心原则：

1. 题目可以手动录入，也可以从面试复盘中沉淀。
2. 题目本身独立存在，不强依赖某一次投递。
3. 题目与面试的关联记录用于表达“这道题曾经在哪些面试中出现过”。
4. 闪卡复习优先服务个人面试准备，不在首期引入复杂学习算法。

关系建议如下：

```text
面试题库 InterviewQuestion 1:N 题目出现记录 QuestionOccurrence
投递记录 ResumeApplication 1:N 题目出现记录 QuestionOccurrence
面试进展 InterviewProgress 1:N 题目出现记录 QuestionOccurrence
面试题库 InterviewQuestion 1:N 闪卡复习记录 FlashcardReview
```

## 5. 适用用户

1. 校招生：需要系统整理八股题、项目追问、算法题和 HR 题。
2. 社招生：需要结合真实面试复盘持续优化知识表达。
3. 高频求职者：需要统计不同公司、岗位和面试轮次中的高频题。
4. 转岗用户：需要按目标岗位标签进行专项复习。

## 6. 核心功能范围

### 6.1 面试题列表

新增“面试题库”页面，作为独立一级模块展示。

页面内采用两个子 tab：

1. 题库列表：用于题目管理、筛选、收藏、查看详情和删除。
2. 闪卡复习：用于按条件抽题并进行卡片式复习。

列表建议支持：

1. 按关键词搜索题目标题和题目内容。
2. 按标签筛选。
3. 按难度筛选。
4. 按题型筛选。
5. 按掌握状态筛选。
6. 按出现次数或更新时间排序。
7. 按是否收藏筛选。
8. 分页展示。

列表卡片或表格建议展示字段：

| 字段 | 说明 |
| --- | --- |
| 题目标题 | 题目的主要识别信息 |
| 标签 | 如 JavaScript、Vue、工程化、算法、项目经历、HR |
| 难度 | 简单 / 中等 / 困难 |
| 题型 | 八股题 / 场景题 / 手写题 / 项目追问 / 行为面试 |
| 掌握状态 | 未复习 / 生疏 / 一般 / 熟练 |
| 出现次数 | 根据关联面试记录统计 |
| 最近复习时间 | 用于判断复习新鲜度 |
| 是否收藏 | 用星星按钮表示收藏状态 |
| 更新时间 | 方便用户识别最近维护内容 |

列表操作建议：

1. 星星收藏按钮：未收藏为空心星，已收藏为实心星。
2. 查看详情。
3. 进入单题闪卡复习。
4. 删除题目。

删除交互要求：

1. 删除题目必须二次确认。
2. 如果题目已经存在面试出现记录，确认文案应提示删除会同步影响关联记录展示。
3. 删除按钮应作为次要或危险操作展示，避免误触。

### 6.2 面试题管理

用户可以新增、编辑、删除面试题。

每道题建议包含以下字段：

| 字段 | 说明 | 是否必填 |
| --- | --- | --- |
| 题目标题 | 列表和闪卡中展示的短标题 | 是 |
| 题目内容 | 具体问题描述，可记录场景和追问 | 是 |
| 答案解析 | 用户整理的答案、思路、易错点和补充材料 | 是 |
| 标签 | 支持多个标签 | 否 |
| 难度 | 简单 / 中等 / 困难 | 是 |
| 题型 | 八股题 / 场景题 / 手写题 / 项目追问 / 行为面试 | 是 |
| 掌握状态 | 未复习 / 生疏 / 一般 / 熟练 | 是 |
| 是否收藏 | 是否加入重点复习 | 否 |
| 来源 | 自己整理 / 面试复盘 / 牛客 / 文章 / 朋友分享等 | 否 |
| 备注 | 其他补充说明 | 否 |

补充说明：

1. 答案解析首期可使用普通多行文本。
2. 后续可扩展 Markdown 编辑和预览能力。
3. 标签首期可采用自由输入或多选，不强制建立复杂标签管理页。

### 6.3 题目标签与分类

标签用于表达知识领域，题型用于表达面试提问方式，二者不要混用。

建议内置标签示例：

1. JavaScript
2. TypeScript
3. Vue
4. React
5. CSS
6. 浏览器
7. 网络
8. 工程化
9. 性能优化
10. 算法
11. 项目经历
12. 系统设计
13. HR 面

建议题型：

1. 八股题
2. 场景题
3. 手写题
4. 算法题
5. 项目追问
6. 行为面试
7. 开放讨论题

### 6.4 题目与面试关联

用户可以将题目关联到某条投递记录，或进一步关联到某一轮面试进展。

关联记录用于表达“这道题在某次真实面试中出现过”，不直接修改题目本身。

题目出现记录建议包含字段：

| 字段 | 说明 | 是否必填 |
| --- | --- | --- |
| 题目 ID | 关联的面试题 | 是 |
| 投递记录 ID | 关联的投递记录 | 是 |
| 面试进展 ID | 关联的一轮面试进展 | 否 |
| 公司名称快照 | 关联时的公司名称 | 是 |
| 岗位名称快照 | 关联时的岗位名称 | 是 |
| 面试阶段快照 | 如一面、二面、HR 面 | 否 |
| 出现时间 | 题目出现或记录时间 | 是 |
| 当时问法 | 面试官实际提问方式或追问点 | 否 |
| 回答表现 | 不会 / 模糊 / 基本答出 / 回答较好 | 否 |
| 备注 | 其他复盘说明 | 否 |

统计规则：

1. 题目的出现次数由题目出现记录数量统计得出。
2. 同一道题可以关联多条投递记录。
3. 同一场面试中同一道题原则上只记录一次，避免重复统计。
4. 删除投递记录时，应明确处理其关联的题目出现记录，避免悬空数据。

### 6.5 从面试复盘沉淀题目

在投递记录详情或面试进展区域，建议提供“沉淀到题库”入口。

主流程：

1. 用户完成一轮面试后，进入投递记录详情。
2. 用户在面试进展或面试小记中记录被问到的问题。
3. 用户选择将某个问题沉淀为新题目，或关联到已有题目。
4. 系统创建题目出现记录，保留公司、岗位、阶段等上下文。
5. 后续题目可在题库中被统一复习和统计。

### 6.6 闪卡复习

闪卡用于轻量复习，首期以“抽题、看题、卡片翻转显示答案、自评掌握情况”为核心。

支持模式建议：

| 模式 | 说明 |
| --- | --- |
| 随机复习 | 从题库中随机抽取题目 |
| 标签专项 | 按一个或多个标签抽题 |
| 难度专项 | 按简单 / 中等 / 困难抽题 |
| 收藏优先 | 优先抽取已收藏题目 |
| 薄弱优先 | 优先抽取生疏、未复习、回答表现较差的题 |
| 高频优先 | 优先抽取真实面试出现次数较高的题 |

闪卡交互建议：

1. 闪卡采用居中的实体卡片视觉，卡片正面展示题目。
2. 用户点击卡片或“翻转”按钮后，卡片以丝滑动画翻转到背面展示答案解析。
3. 翻转动画应有明确的 3D 翻面感，避免直接切换内容。
4. 用户可以通过左右滑动卡片切换上一题或下一题。
5. 滑动切换应有连续位移、轻微缩放或透明度变化，形成自然的卡片过渡效果。
6. 用户根据回忆情况选择“不会 / 模糊 / 掌握”。
7. 系统更新题目的掌握状态和最近复习时间。
8. 系统记录本次复习结果，供后续统计使用。

动画体验要求：

1. 翻转和滑动过程应保持流畅，不出现明显卡顿、跳帧或内容闪烁。
2. 卡片在翻转期间应避免正反面文字同时可见。
3. 滑动到下一题后，卡片应回到正面题目状态。
4. 动画时长应适中，建议 250ms-400ms。

### 6.7 复习记录

每次闪卡自评后生成复习记录。

复习记录建议包含字段：

| 字段 | 说明 |
| --- | --- |
| 题目 ID | 被复习的题目 |
| 复习时间 | 用户完成自评的时间 |
| 复习结果 | 不会 / 模糊 / 掌握 |
| 复习模式 | 随机 / 标签专项 / 薄弱优先等 |

首期不要求实现复杂间隔重复算法，但数据结构应便于后续扩展。

## 7. 核心业务流程

### 7.1 手动录入题目流程

1. 用户进入面试题库列表页。
2. 点击新增题目。
3. 填写题目标题、题目内容、答案解析、标签、难度、题型和掌握状态。
4. 保存后返回题库列表或进入题目详情页。

### 7.2 面试后沉淀题目流程

1. 用户进入某条投递记录详情页。
2. 用户查看或新增某一轮面试进展。
3. 用户点击“关联题目”或“沉淀到题库”。
4. 用户选择已有题目，或创建新题目。
5. 系统保存题目出现记录。
6. 题库中的出现次数和面试上下文同步更新。

### 7.3 闪卡复习流程

1. 用户进入闪卡复习页。
2. 用户选择复习模式和抽题范围。
3. 系统展示第一张题目卡片正面。
4. 用户思考后点击卡片翻转查看答案。
5. 用户选择不会、模糊或掌握。
6. 用户通过滑动或按钮切换到下一张卡片。
7. 系统记录复习结果并进入下一题。
8. 复习结束后展示本次复习概览。

## 8. 页面建议

### 8.1 左侧导航

建议新增“面试题库”一级菜单，不挂在“简历管理”二级菜单下。

原因：

1. 面试题库是独立知识资产，不完全从属于某份简历。
2. 题库会同时服务多个岗位、多个投递和多场面试。
3. 用户面试前复习时应能快速进入。

### 8.2 面试题列表页

建议采用列表或紧凑卡片形式，整体信息密度略高于岗位卡片。

页面区域建议：

1. 顶部子 tab：题库列表 / 闪卡复习。
2. 题库列表 tab 内展示搜索与筛选区。
3. 题目列表区。
4. 新增题目入口。
5. 每条题目展示收藏星星按钮和删除按钮。

收藏按钮视觉建议：

1. 未收藏状态显示空心星。
2. 已收藏状态显示实心星，并使用轻量强调色。
3. 收藏按钮应放在列表项右侧的稳定位置，避免影响标题阅读。

删除按钮视觉建议：

1. 删除按钮放在列表项操作区。
2. 删除应使用危险色或文字按钮样式。
3. 点击后弹出确认，不直接删除。

### 8.3 面试题详情页

详情页建议包含：

1. 题目基础信息区：标题、标签、难度、题型、掌握状态。
2. 题目内容区：完整问题描述。
3. 答案解析区：答案、思路、注意点。
4. 出现记录区：展示该题在哪些公司、岗位和面试阶段出现过。
5. 复习记录区：展示最近复习时间和掌握变化。

### 8.4 闪卡复习页

闪卡复习作为“面试题库”模块下的第二个子 tab 展示，应尽量减少干扰，突出卡片和答案。

建议状态：

1. 正面状态：展示题目内容、标签、难度和当前进度。
2. 翻转中状态：卡片执行 3D 翻转动画。
3. 背面状态：展示答案解析、自评按钮和翻回题目入口。
4. 滑动切换状态：卡片向左或向右滑出，下一张卡片自然进入。
5. 复习完成：展示本次复习数量、不会数量、模糊数量、掌握数量。

## 9. 数据结构建议

### 9.1 面试题实体

```ts
interface InterviewQuestion {
  id: string
  title: string
  content: string
  answerAnalysis: string
  tags: string[]
  difficulty: 'easy' | 'medium' | 'hard'
  questionType: 'knowledge' | 'scenario' | 'coding' | 'algorithm' | 'project' | 'behavior' | 'discussion'
  masteryStatus: 'unreviewed' | 'weak' | 'normal' | 'mastered'
  isFavorite: boolean
  source?: string
  occurrenceCount: number
  lastReviewedAt?: string
  remark?: string
  createdAt: string
  updatedAt: string
}
```

### 9.2 题目出现记录实体

```ts
interface QuestionOccurrence {
  id: string
  questionId: string
  applicationId: string
  interviewProgressId?: string
  companyNameSnapshot: string
  jobTitleSnapshot: string
  interviewStageSnapshot?: string
  occurredAt: string
  actualQuestion?: string
  answerPerformance?: 'failed' | 'vague' | 'answered' | 'good'
  note?: string
  createdAt: string
  updatedAt: string
}
```

### 9.3 闪卡复习记录实体

```ts
interface FlashcardReview {
  id: string
  questionId: string
  reviewedAt: string
  result: 'failed' | 'vague' | 'mastered'
  reviewMode: 'random' | 'tag' | 'difficulty' | 'favorite_first' | 'weak_first' | 'frequent_first'
  createdAt: string
}
```

## 11. 核心规则

1. 面试题可以不关联任何投递记录而独立存在。
2. 题目出现记录必须关联到一条投递记录。
3. 题目出现记录可以选择性关联到某一条面试进展。
4. 题目的出现次数应由题目出现记录统计得出。
5. 闪卡复习结果应更新题目的掌握状态和最近复习时间。
6. 删除题目时，应同步处理该题目的出现记录和复习记录。
7. 删除投递记录或面试进展时，应避免题目出现记录变成不可解释的悬空数据。
8. 标签、难度、题型、掌握状态应在列表筛选和闪卡抽题中保持一致。
9. 题库列表和闪卡复习必须是同一模块下两个不同子 tab。
10. 收藏状态应可在题库列表中直接切换，并能作为闪卡抽题条件。
11. 闪卡翻转后切换到下一题时，应自动恢复为正面题目状态。

## 12. 非功能要求

1. 页面风格保持当前 Element UI 后台系统风格。
2. 列表页应具备加载态、空状态和无搜索结果状态。
3. 题目内容和答案解析应支持较长文本输入。
4. 闪卡页应保证阅读区域清晰，避免过多统计信息干扰复习。
5. 闪卡翻转和滑动动画应保持丝滑、稳定和可预期。
6. 数据结构应便于后续扩展间隔重复、AI 答案优化和复习统计。

## 13. 验收标准

1. 用户可以成功新增、编辑、删除面试题。
2. 用户可以为题目设置标签、难度、题型和掌握状态。
3. 用户可以在题库列表中按关键词、标签、难度、题型和掌握状态筛选题目。
4. 用户可以在题库列表中收藏或取消收藏题目。
5. 用户可以在题库列表中删除题目，删除前必须二次确认。
6. 用户可以为题目维护答案解析。
7. 用户可以将题目关联到某条投递记录或某一轮面试进展。
8. 系统可以展示题目的真实面试出现次数和出现记录。
9. 题库列表和闪卡复习作为两个不同子 tab 展示。
10. 用户可以进入闪卡复习 tab 随机抽题。
11. 用户可以按标签、难度、收藏优先、薄弱优先或高频优先进行闪卡复习。
12. 闪卡以卡片形式展示，点击后通过翻转动画显示答案。
13. 用户可以通过滑动卡片切换上一题或下一题。
14. 用户在查看答案后可以标记不会、模糊或掌握。
15. 系统可以记录复习结果，并更新题目的掌握状态和最近复习时间。
16. 面试题库入口作为独立一级模块展示。

## 14. 后续可拓展方向

1. 增加间隔重复算法，根据掌握状态和复习历史自动安排下次复习。
2. 增加 Markdown 答案编辑和预览。
3. 增加 AI 辅助生成答案解析、追问点和回答框架。
4. 增加从面试小记文本中半自动提取题目的能力。
5. 增加按公司、岗位、标签维度的高频题统计看板。
6. 增加面试前冲刺模式，根据目标岗位标签自动组卷。
7. 增加题目导入导出能力，方便备份和迁移。
