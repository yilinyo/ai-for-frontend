# 岗位库与岗位关联投递功能 PRD

## 1. 基本信息

| 项目名称 | 文档版本 | 创建日期 | 适用平台 |
| --- | --- | --- | --- |
| 岗位库与岗位关联投递功能 | V0.1 | 2026-04-26 | Web 后台 |

## 2. 需求背景

当前系统已经具备简历版本管理、投递记录管理和面试进展管理能力，但在实际求职过程中，用户往往不是先投递再整理信息，而是会先收藏岗位、记录岗位详情、对比多个岗位，再决定是否投递。

如果岗位信息只存在于投递记录里，会有几个问题：

1. 用户无法先沉淀岗位信息，只能等投递时再录入。
2. 同一个岗位如果后续再次投递或换用新的简历版本，岗位信息会重复录入。
3. 岗位来源链接、岗位要求、公司介绍等信息无法作为独立资产沉淀。
4. 用户缺少一个统一的“岗位池”来管理正在关注的岗位。

因此需要新增“岗位库”能力，将岗位作为独立对象先沉淀下来。岗位支持手动录入，也支持输入岗位来源链接后由后端解析并返回结构化 JSON 数据。前端只负责调用解析接口和展示结果，不关心具体解析逻辑。后续创建投递记录时，可直接从岗位库中选择已有岗位进行关联。

## 3. 需求目标

1. 新增独立的岗位库模块，用于统一管理岗位信息。
2. 支持岗位手动录入和岗位链接自动解析导入两种录入方式。
3. 支持从岗位库直接跳转到原始岗位链接，如 Boss、官网、牛客等。
4. 支持在投递记录创建时直接选择已有岗位库数据进行关联，避免重复填写岗位信息。
5. 保证岗位详情字段与投递记录中的岗位信息字段保持一致。

## 4. 方案定位

本次采用 `C` 方案：

1. 岗位作为独立岗位库存在。
2. 岗位先独立沉淀，再在投递时与简历版本生成投递记录关联。
3. 一条岗位记录可以在不同时间、不同简历版本下被多次投递使用。

关系建议如下：

```text
岗位库 JobPosting 1:N 投递记录 ResumeApplication
简历版本 ResumeVersion 1:N 投递记录 ResumeApplication
投递记录 ResumeApplication 1:N 面试进展 InterviewProgress
```

## 5. 核心功能范围

### 5.1 岗位列表

新增“岗位列表”页面，整体风格与当前“简历仓库列表”保持一致，采用：

1. 顶部搜索筛选区。
2. 卡片式岗位列表。
3. 分页展示。
4. 创建岗位按钮。

页面入口建议：

1. 左侧导航新增“岗位列表”一级菜单。
2. 保持与当前后台项目风格一致，不做复杂多 tab 工作台样式。

### 5.2 岗位卡片

岗位卡片保留主要信息，避免列表过重。建议卡片展示字段：

| 字段 | 说明 |
| --- | --- |
| 公司名称 | 主要识别信息 |
| 岗位名称 | 主要识别信息 |
| base | 工作地点 |
| 薪资范围 | 快速判断 |
| 来源平台 | Boss / 官网 / 牛客 / 内推等 |
| 岗位状态 | 待投递 / 已投递 / 不合适 / 已关闭 |
| 创建时间或收藏时间 | 方便回顾 |

卡片操作建议：

1. 查看详情
2. 编辑岗位
3. 删除岗位
4. 查看原岗位

其中“查看原岗位”可直接跳转岗位来源链接。

### 5.3 岗位创建方式

岗位创建支持两种方式：

#### 方式一：手动录入

用户直接填写岗位信息并保存。

#### 方式二：输入链接自动解析

用户输入岗位来源链接，前端调用解析接口，后端返回岗位结构化 JSON 数据，前端将其回填到表单中供用户确认和编辑。

本期前端要求：

1. 提供“输入链接并解析”的交互入口。
2. 前端通过 Mock 接口模拟“解析成功后返回岗位 JSON”。
3. 前端不处理网页抓取逻辑，只消费解析结果。

后端预期能力：

1. 根据岗位来源链接解析页面。
2. 返回统一格式的岗位结构化数据。

### 5.4 岗位详情

岗位详情页字段应与投递记录中使用的岗位字段完全一致，避免字段定义不统一。

建议岗位详情字段：

| 字段 | 说明 | 是否必填 |
| --- | --- | --- |
| 公司名称 | 岗位所属公司 | 是 |
| 公司介绍 | 公司业务方向、规模、团队信息 | 否 |
| 岗位名称 | 岗位标题 | 是 |
| 岗位要求 | JD 摘要、要求、加分项 | 否 |
| base | 工作地点 | 否 |
| 薪资范围 | 如 20k-30k x 14 | 否 |
| 来源平台 | Boss、官网、牛客、内推等 | 否 |
| 来源链接 | 原始岗位 URL | 否 |
| 岗位状态 | 待投递 / 已投递 / 不合适 / 已关闭 | 是 |
| 备注 | 用户补充说明 | 否 |

详情页操作建议：

1. 编辑岗位
2. 打开原岗位链接
3. 基于该岗位创建投递记录
4. 查看该岗位关联的历史投递记录

### 5.5 岗位与投递记录关联

后续创建投递记录时，支持两种方式：

1. 直接选择岗位库中已有岗位。
2. 不选择岗位，继续手动填写岗位信息。

但本期推荐主路径为：

1. 先沉淀岗位到岗位库。
2. 创建投递记录时选择已有岗位。
3. 系统自动带出公司名称、岗位名称、岗位要求、base、薪资范围、来源平台等字段。
4. 用户只需补充简历版本、投递日期、当前状态和面试信息。

建议新增字段：

| 字段 | 说明 |
| --- | --- |
| jobPostingId | 关联岗位库主键 |

当投递记录关联了岗位库后：

1. 投递记录应保留岗位快照字段，避免岗位库后续修改影响历史投递语义。
2. 同时保留 `jobPostingId`，用于关联和跳转。

## 6. 页面设计要求

### 6.1 岗位列表页

风格参考当前简历仓库列表页：

1. 顶部搜索栏支持按公司名称或岗位名称搜索。
2. 支持按来源平台筛选。
3. 支持按岗位状态筛选。
4. 卡片宫格布局与当前仓库卡片风格保持一致。

### 6.2 岗位详情页

风格参考当前“简历版本查看页”和“投递详情页”的信息密度：

1. 顶部展示公司名称、岗位名称、状态标签。
2. 中间采用“岗位基础信息 + 操作按钮”合并区域，避免拆成两个信息块。
3. 在同一区域内同时展示岗位核心字段和主要操作按钮，如编辑岗位、查看原岗位、创建投递记录。
4. 下方展示关联投递记录列表。

### 6.3 视觉一致性

需保持以下一致性：

1. 使用 Element UI 卡片、按钮、Tag、输入框体系。
2. 列表页与简历仓库列表页视觉结构一致。
3. 颜色风格保持当前系统样式，不单独引入新的重主题风格。

## 7. 核心业务流程

### 7.1 岗位沉淀流程

1. 用户进入岗位列表页。
2. 选择手动创建岗位，或输入岗位链接进行解析。
3. 系统生成岗位记录并进入岗位详情页。
4. 用户可补充备注和状态。

### 7.2 从岗位到投递流程

1. 用户在岗位详情页点击“创建投递记录”。
2. 系统进入投递记录创建页。
3. 自动带出岗位字段。
4. 用户选择简历版本并补充投递时间、状态等信息。
5. 保存后生成一条与岗位库关联的投递记录。

## 8. 数据结构建议

### 8.1 岗位库实体

```ts
interface JobPosting {
  id: string
  companyName: string
  companyIntro?: string
  jobTitle: string
  jobRequirements?: string
  base?: string
  salaryRange?: string
  sourcePlatform?: string
  sourceUrl?: string
  status: 'pending' | 'applied' | 'not_fit' | 'closed'
  remark?: string
  createdAt: string
  updatedAt: string
}
```

### 8.2 解析接口建议

```ts
POST /api/job-postings/parse
```

请求参数：

```json
{
  "url": "https://www.zhipin.com/job_detail/xxx"
}
```

返回示例：

```json
{
  "companyName": "腾讯科技",
  "companyIntro": "互联网科技公司",
  "jobTitle": "前端开发工程师",
  "jobRequirements": "熟悉 Vue / React ...",
  "base": "深圳",
  "salaryRange": "25k-35k x 16",
  "sourcePlatform": "Boss",
  "sourceUrl": "https://..."
}
```

## 9. 验收标准

1. 系统存在独立“岗位列表”入口。
2. 岗位列表页采用与简历仓库列表一致的卡片风格。
3. 用户可手动创建岗位。
4. 用户可输入链接并模拟解析岗位信息。
5. 岗位卡片和详情页可跳转来源链接。
6. 岗位详情字段与投递记录中的岗位信息字段保持一致。
7. 创建投递记录时可直接选择岗位库中的岗位进行关联。

## 10. 当前版本边界

本期包含：

1. 岗位库基础 CRUD
2. 岗位链接解析 Mock 流程
3. 岗位与投递记录的关联入口
4. 岗位列表与详情页

本期不包含：

1. 真实网页解析能力
2. 批量导入岗位
3. 岗位推荐和智能匹配
4. 多平台账号同步

## 11. 默认设计假设

本版 PRD 采用以下默认假设：

1. 岗位列表作为左侧导航独立入口存在。
2. 岗位库先独立存在，再与投递记录关联。
3. 岗位卡片与简历仓库卡片保持同类风格。
4. 原岗位链接在卡片和详情页都提供跳转入口。
