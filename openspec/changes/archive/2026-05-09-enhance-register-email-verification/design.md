## Context

现有注册页已经有用户名、密码、确认密码和邮箱字段，但邮箱是选填，且没有验证码验证。用户模型和 Mock 注册逻辑会把邮箱写入新用户资料，个人信息页也会读取当前账号的 `email` 字段，因此注册邮箱一旦被正确写入，后续可自然回显到个人信息页。

本次变更跨越模型、API、Mock 和注册页交互，但不引入真实邮件服务。实现上需要保持 Vue 2 + TypeScript + Element UI 的现有风格，并继续采用 Mock-first。

## Goals / Non-Goals

**Goals:**

- 注册邮箱必填并校验格式。
- 支持发送邮箱验证码，并在注册时校验验证码。
- 验证码必须与邮箱绑定，并具备有效期。
- 已存在用户名和已存在邮箱都应拦截注册。
- 注册成功后邮箱写入账号个人资料，登录后个人信息页可回显。
- 当前阶段通过 Mock 模拟验证码发送和校验。
- 登录页提供清晰的注册入口，避免用户只能通过直接访问注册路由完成注册。
- 登录页和注册页采用一致的企业级认证页布局，提升进入系统前的专业感。

**Non-Goals:**

- 不接入真实 SMTP、短信或第三方验证码服务。
- 不新增邮箱登录、找回密码、修改邮箱验证流程。
- 不新增图形验证码、IP 限流、设备指纹等复杂风控。
- 不调整个人信息页邮箱编辑策略。

## Decisions

验证码发送接口独立于注册接口。
原因：用户需要先获取验证码，再提交注册。独立接口更贴近真实后端设计，也方便未来替换为真实邮件服务。
备选方案：注册接口内部自动发送验证码，但这无法完成“先验证再注册”的交互闭环。

Mock 验证码固定为 `123456`。
原因：本需求主要针对前端注册页设计，固定验证码可以降低 Mock 复杂度并方便本地演示。Mock 可保留基础邮箱绑定和有效期校验，但不在规格中展开过多后端场景。
备选方案：随机生成验证码更贴近真实场景，但会增加前端演示和自动化验证成本。

发送验证码成功后前端启动倒计时。
原因：避免重复点击发送，并给用户清晰反馈。只有发送成功后才进入倒计时，发送失败不应锁定按钮。
备选方案：不做倒计时，交互简单但容易重复请求。

注册邮箱作为个人资料邮箱来源。
原因：注册成功时创建用户资料，邮箱应随用户资料一起保存；个人信息页继续通过当前账号资料读取 `email`，无需额外联动接口。
备选方案：注册后再要求用户到个人信息页填写邮箱，会重复输入且不符合验证码验证目标。

登录页显式暴露注册入口。
原因：本次注册能力增强后，登录页需要给新用户提供可发现的入口，形成完整的登录 / 注册闭环。
备选方案：仅保留直接访问注册路由，但入口不可见会影响新用户使用。

登录页和注册页统一采用企业级认证页布局。
原因：当前功能面向求职流程管理系统，登录注册页是用户进入系统的第一屏，需要比普通表单页更有品牌感和可信度。布局可参考知乎类登录注册页：一侧承载系统名称、价值说明或插画氛围，另一侧承载表单卡片与登录 / 注册切换入口。
备选方案：仅在现有布局上增加验证码字段，改动较小但视觉质感和入口完整性不足。

## Risks / Trade-offs

- [Mock 返回验证码与真实环境不同] → Mitigation: 明确这是 Mock-first 行为，真实后端接入时移除 `mockCode` 返回。
- [前端交互与后端真实校验边界混淆] → Mitigation: 本 change 以注册页交互设计为主，Mock 只保证流程可验证。
- [邮箱修改后验证码状态混乱] → Mitigation: 前端邮箱变更时清空验证码，并要求用户重新获取或填写验证码。
- [视觉升级范围过大] → Mitigation: 仅调整登录页和注册页的认证页布局，不扩展到登录后的业务页面。
