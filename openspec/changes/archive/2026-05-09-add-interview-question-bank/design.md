## Context

当前项目基于 Vue 2、TypeScript、Element UI、Vuex 和 Express Mock API，业务上已经形成“简历仓库 -> 简历版本 -> 投递记录 -> 面试进展”的求职流程链路。现有面试复盘主要依赖投递记录中的 `interviewNotes` 和面试进展 `note` 字段，适合记录自由文本，但不适合沉淀可复习、可统计、可按标签筛选的题目资产。

本次变更将在现有分层约定下新增面试题库模块，优先遵循 `src/models/`、`src/api/`、`mock/`、`src/store/modules/`、`src/router/modules/`、`src/views/`、`src/components/` 的结构。该模块需要与投递记录和面试进展发生关联，但面试题本身应作为独立知识对象存在，不从属于某一条投递记录。

需求文档与视觉稿位于：

- `docs/PRD文档/面试题库方案/PRD_面试题库与闪卡复习_简版.md`
- `docs/PRD文档/面试题库方案/VISUAL_面试题库与闪卡复习_mockup.svg`

## Goals / Non-Goals

**Goals:**

- 新增面试题库一级模块，并在左侧菜单下提供“题库列表”和“闪卡复习”两个二级入口。
- 将题库列表和闪卡复习拆分为两个 OpenSpec 能力：`interview-question-library` 与 `interview-flashcard-review`。
- 建立面试题、面试提问记录、闪卡复习记录三类核心数据模型。
- 支持题目 CRUD、标签/难度/题型/掌握状态筛选、收藏切换和删除确认。
- 支持题目内容和答案解析使用 Markdown 格式，并在编辑时粘贴图片附件生成 Markdown 图片。
- 支持题目详情页双击题目内容或答案解析进入编辑，作为“编辑题目”按钮之外的轻量入口。
- 支持将题目关联到投递记录或具体面试进展，统计真实面试提问频率。
- 支持从投递记录详情或面试进展中沉淀题目，形成复盘到复习的入口。
- 实现卡片式闪卡复习，支持点击翻转显示答案、左右滑动切换题目、自评并记录复习结果。
- 保持页面风格与现有 Element UI 后台一致，避免引入新的视觉体系。

**Non-Goals:**

- 不实现复杂间隔重复算法，只保留可扩展的数据结构和基础复习结果记录。
- 不实现 AI 生成答案、自动提取题目、外部题库导入导出。
- 不引入新的 UI 框架或大型动画库。
- 不改造既有投递记录和面试进展的核心状态流转规则。
- 不将题目标签做成独立全量标签管理系统，首期采用题目字段内的标签集合。

## Decisions

### 1. 面试题库作为独立一级模块，题库列表和闪卡复习作为左侧二级入口

新增面试题库相关路由，并在侧边栏作为一级菜单展示。参照权限测试页的导航方式，在“面试题库”下提供“题库列表”和“闪卡复习”两个二级菜单；内容区直接展示对应页面，不再额外使用 tab 切换。能力边界上仍拆成 `interview-question-library` 和 `interview-flashcard-review`。

理由：

- 面试题服务多个岗位、投递和面试，不应挂在简历管理二级菜单下。
- 用户面试前复习需要快速进入，不应先进入某条投递记录。
- 题库列表负责知识资产管理，闪卡复习负责训练体验，拆成两个能力更利于规格、任务和验收独立闭环。
- 两个能力共享题目数据，放在同一一级菜单下的两个二级入口更清晰，也能节省内容区 tab 占用的纵向空间。

备选方案：

- 放在投递详情页内：入口贴近复盘，但不适合独立复习和跨公司统计。
- 拆成两个一级菜单：导航过重，且题库管理与复习的数据边界并不独立。
- 合并成一个 `interview-question-bank` 能力：文档更少，但会把题库管理和复习交互混在同一验收口径里。

### 1.1 题库列表整行点击进入详情

题库列表每一行都作为详情入口，用户点击列表项任意非操作区域即可进入题目详情页。收藏、编辑、关联提问、删除等操作按钮需要阻止事件冒泡，避免触发整行跳转。

理由：

- 题库列表的主行为是查看题目详情，整行点击比只点“详情”按钮更符合后台表格浏览习惯。
- 操作按钮行为必须明确，不应因为整行点击导致误跳转。

### 2. 题目与面试提问上下文拆分建模

核心模型拆为：

- `InterviewQuestion`：题目本体，包含标题、内容、答案解析、标签、难度、题型、掌握状态、收藏状态、创建时间、修改时间、最近复习时间等。
- `QuestionOccurrence`：题目在真实面试中的提问记录，关联题目、投递记录和可选面试进展，并保存公司、岗位、阶段等快照。内部类型名沿用 `QuestionOccurrence`，用户界面统一展示为“面试提问记录”。
- `FlashcardReview`：用户一次闪卡自评结果，关联题目并记录复习模式、结果和时间。

理由：

- 同一道题可能在多场面试中被问到，不能把公司和阶段直接写进题目本体。
- 投递记录或岗位后续编辑不应改变历史提问语义，因此面试提问记录需要保存快照字段。
- 复习记录独立保存，可以支持后续间隔重复、趋势统计和面试前冲刺。
- 创建时间用于识别题目沉淀时间，修改时间用于追踪内容维护；列表页为保持表格可读性只展示创建时间，详情页展示两者。

备选方案：

- 只在题目上保存 `occurrenceCount`：实现简单，但丢失真实面试上下文。
- 将题目直接嵌入面试进展：复盘方便，但无法形成跨面试题库。

### 3. 提问次数由面试提问记录统计，题目列表可使用派生字段展示

Mock 层可以在返回题目列表时根据 `QuestionOccurrence` 计算 `occurrenceCount`，也可以在内存数据中维护派生字段用于展示。前端展示时以接口返回值为准，不在页面层自行遍历所有面试提问记录计算。

理由：

- 列表页需要排序和筛选，提问次数应由 API 层统一提供。
- 前端页面保持简单，避免不同页面计算口径不一致。

备选方案：

- Vuex 层自行聚合：减少接口复杂度，但会把业务统计逻辑散落到前端状态层。

### 4. 删除题目采用二次确认并级联处理关联展示数据

删除题目时，如果存在面试提问记录或复习记录，交互上必须提示删除会影响关联记录展示。Mock 实现中删除题目后应同步删除该题目的面试提问记录和复习记录，避免悬空数据。

理由：

- 题目是面试提问记录和复习记录的主对象，删除后保留子记录会造成不可解释的数据。
- 二次确认可以降低误删重点题目的风险。

备选方案：

- 禁止删除有关联记录的题目：数据更保守，但个人工具场景下会让清理题库变得笨重。
- 软删除：利于恢复，但首期会增加列表筛选和 Mock 数据复杂度。

### 5. 收藏状态直接保存在题目本体

`isFavorite` 作为 `InterviewQuestion` 字段，由题库列表星星按钮直接切换，并可作为闪卡抽题条件。

理由：

- 收藏是用户对题目本身的主观标记，不依赖某次复习或面试。
- 列表和闪卡入口都需要快速读取收藏状态。

备选方案：

- 单独建收藏表：适合多用户多集合场景，但当前个人题库首期不需要。

### 6. 题目内容与答案解析使用 Markdown 展示和编辑

题目内容和答案解析仍保存在 `InterviewQuestion.content` 与 `InterviewQuestion.answerAnalysis` 字符串字段中，但字段语义升级为 Markdown 文本。详情页、列表摘要和闪卡展示应渲染 Markdown 的可读结果；编辑弹窗保留表单结构，在题目内容和答案解析输入区支持粘贴图片附件，前端将图片转为本地 Data URL 并插入 Markdown 图片语法。

理由：

- Markdown 能兼容纯文本、代码块、列表、链接、图片等面试复盘常用格式。
- 不引入新的后端附件服务时，Data URL 可以满足 Mock 阶段的图片粘贴和预览验证。
- 双击内容或答案解析直接打开编辑弹窗，减少查看详情后的修改路径成本。

备选方案：

- 使用富文本 HTML 编辑：展示效果直接，但会增加清洗和存储复杂度。
- 新增独立附件模型：更适合生产文件存储，但当前前端 Mock 阶段会扩大范围。

### 7. 闪卡能力只消费题库能力提供的题目数据

闪卡复习不直接维护题目 CRUD，也不直接创建面试提问记录。它通过抽题接口消费题库题目和题目派生状态，并在自评后写入 `FlashcardReview`，同时更新题目掌握状态与最近复习时间。

理由：

- 保持题库管理和复习训练的职责分离。
- 后续可以单独增强闪卡复习算法，而不影响题库列表、详情和面试提问记录。

备选方案：

- 闪卡页面直接修改完整题目对象：实现更直接，但会让复习能力侵入题库管理逻辑。

### 8. 闪卡动画使用组件内状态和 CSS Transition 实现

闪卡复习页维护当前题目索引、是否翻转、滑动方向、动画中状态和自评结果。卡片翻转使用 CSS `transform: rotateY(...)`、`backface-visibility` 和 `transition` 实现；滑动切换使用位移、透明度和轻微缩放的 class 状态实现。

理由：

- 不引入新动画库，符合当前项目依赖控制。
- CSS transition 足以实现 250ms-400ms 的翻转和滑动体验。
- Vue 2 组件内状态可以稳定控制“下一题回到正面”的规则。

备选方案：

- 引入手势或动画库：交互更强，但会增加依赖和调试成本。
- 只用按钮切换不做滑动：实现简单，但不满足本次视觉和体验要求。

### 9. 投递详情页只增加轻量入口，不重做面试进展结构

在投递记录详情页或面试进展区域增加“关联题目 / 沉淀到题库”入口。入口可以打开弹窗，让用户选择已有题目或创建新题目，并保存 `QuestionOccurrence` 面试提问记录。界面文案避免使用“出现记录”，统一称为“面试提问记录”或“提问次数”。不改变当前面试进展创建、编辑、删除和 `currentStatus` 同步逻辑。

理由：

- 保持面试进展模块稳定，降低对既有流程的影响。
- 题库沉淀是复盘增强能力，不应侵入当前状态流转。

备选方案：

- 将面试进展 note 拆成结构化题目列表：更彻底，但会扩大改造范围并影响现有复盘输入。

## Risks / Trade-offs

- [题库模块范围偏大] -> 首期聚焦题目管理、真实面试提问记录和基础闪卡，不做 AI、导入导出和间隔重复。
- [题目删除导致历史复盘信息减少] -> 删除前提示关联影响，并在任务中覆盖有关联记录的删除验证。
- [闪卡滑动与翻转状态冲突] -> 组件状态中区分 `isFlipped`、`isAnimating`、`swipeDirection`，动画期间禁用重复触发。
- [移动端或窄屏下卡片文字溢出] -> 闪卡卡片设置固定响应式宽度、最大高度和答案滚动区域，保证文字不遮挡操作按钮。
- [提问次数统计口径不一致] -> 统一由 Mock/API 返回，页面只消费结果。
- [投递详情入口增加页面复杂度] -> 使用弹窗或轻量按钮承载，不改变原有面试进展时间线和状态节点结构。

## Migration Plan

1. 新增 Mock 内存数据和接口，不迁移现有数据。
2. 提供若干示例题目、面试提问记录和复习记录，覆盖收藏、高频、薄弱、不同标签等状态。
3. 现有投递记录和面试进展数据保持不变。
4. 如需回滚，可移除新增路由、菜单、Mock 接口和题库相关 store，不影响简历、岗位、投递和面试进展主链路。

## Open Questions

- 题目详情页首期是否需要展示复习记录列表，还是只展示最近复习时间和累计结果摘要？
- 闪卡滑动是否仅支持桌面鼠标拖拽和触控事件，还是首期只提供按钮切换加基础拖拽视觉？
