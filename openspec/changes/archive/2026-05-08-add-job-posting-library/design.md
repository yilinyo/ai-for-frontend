## 背景

本项目当前是一个基于 Vue 2、TypeScript 和 Element UI 的后台应用，已具备简历仓库、简历版本、投递记录和面试进展管理能力。现有业务结构里，岗位信息主要包含在投递记录中，但尚未抽象为独立岗位库实体。

根据已确认的方案，本次功能采用独立岗位库设计：

1. 岗位先独立沉淀到岗位库。
2. 用户可以手动录入岗位，也可以输入原始岗位链接，由解析接口返回结构化 JSON 后回填表单。
3. 岗位详情页中可直接发起“创建投递记录”。
4. 后续投递记录可通过 `jobPostingId` 关联岗位库，并同时保留岗位快照字段。

实现上需要保持与当前项目架构一致，继续采用 `models -> api -> mock -> store -> route -> view` 分层，并复用当前简历仓库列表页的卡片风格。

## 目标 / 非目标

**目标：**

- 新增岗位库模块，支持岗位信息的独立沉淀和管理。
- 支持岗位手动录入与链接解析导入两种创建方式。
- 新增岗位列表页与岗位详情页，并保持与现有后台风格一致。
- 将岗位列表作为与“简历管理”同级的一级导航入口，而不是其二级菜单。
- 支持从岗位详情页直接跳转到投递记录创建流程。
- 支持投递记录选择已有岗位并自动带出岗位字段。

**非目标：**

- 本次不实现真实网页抓取或解析逻辑。
- 本次不实现多平台账号同步。
- 本次不实现岗位推荐、智能匹配或批量导入。
- 本次不实现复杂工作台式多 tab 布局，仍采用当前后台导航结构。

## 关键设计决策

采用独立岗位库而不是仅在投递记录中维护岗位字段。
原因：用户在求职过程中通常会先收集岗位、再决定投递。独立岗位库更符合“岗位池”管理心智，也方便多次投递复用。
备选方案：
- 仅在投递记录中维护岗位信息：实现简单，但不利于岗位沉淀和复用。
- 从投递记录中反向生成岗位列表：信息不完整，且会被投递时机绑定。

岗位列表采用一级导航，而不是挂在“简历管理”下。
原因：岗位库与简历仓库、投递记录一样，已经成为独立一级模块。如果继续放在“简历管理”二级菜单下，会弱化岗位池的独立定位，也不符合最终确认的导航层级。
备选方案：
- 将岗位列表挂在“简历管理”下作为二级菜单：实现上简单，但信息架构不符合已确认口径。

岗位详情页采用“岗位基础信息与操作”合并区域。
原因：用户已明确希望不要拆成两个区域。将岗位字段和核心操作按钮放在同一信息块中，信息更集中，操作路径更短。
备选方案：
- 单独拆出操作区：结构清晰，但页面更碎。

岗位编辑采用弹窗而不是独立编辑页。
原因：岗位字段虽然不少，但复杂度仍低于投递记录和面试进展，适合在列表页或详情页以弹窗方式快速修改。
备选方案：
- 独立编辑页：更适合超长表单，但当前阶段显得过重。

“查看原岗位”只负责打开原始来源页面，不承担重新解析职责。
原因：该操作的目标是让用户快速回到 Boss、官网、牛客等来源页面查看最新 JD 或招聘状态。
备选方案：
- 将其兼作重新解析入口：语义混乱，不利于后续扩展。

岗位卡片的尺寸、内容区高度和留白需要与简历仓库卡片保持一致。
原因：岗位列表与简历列表都采用卡片宫格，如果卡片高度和留白节奏不一致，会造成并排浏览时的视觉跳动。岗位卡片需要通过固定内容区高度、贴底 footer 和容器留白控制，保持整体观感一致。
备选方案：
- 允许岗位卡片按内容自然撑开：实现简单，但会导致不同卡片高度不一致。

岗位解析采用前后端分离职责。
原因：前端只需要发起解析请求并消费统一 JSON，解析逻辑应由后端负责。当前阶段先以 Mock 返回解析结果，保证前端交互和数据流先落地。
备选方案：
- 前端自行抓取和解析：不可控，也不符合架构边界。

投递记录关联岗位时，同时保留 `jobPostingId` 和岗位快照字段。
原因：如果只保存 `jobPostingId`，岗位库后续被编辑时可能影响历史投递语义；保留快照后，投递记录能够稳定回溯当时的岗位信息。
备选方案：
- 只关联岗位主键：历史稳定性不足。

## 数据模型设计

### 岗位库实体

建议结构：

```ts
interface JobPosting {
  id: string
  companyName: string
  companyIntro?: string
  jobTitle: string
  jobRequirements?: string
  base?: string
  salaryRange?: string
  sourcePlatform?: string
  sourceUrl?: string
  status: JobPostingStatus
  remark?: string
  createdAt: string
  updatedAt: string
}
```

### 岗位状态枚举

建议状态：

- `pending`：待投递
- `applied`：已投递
- `not_fit`：不合适
- `closed`：已关闭

这组状态描述的是“岗位本身是否还值得继续跟进”，而不是具体某次投递的面试流程状态。

### 投递记录扩展

在现有 `ResumeApplication` 基础上增加：

```ts
jobPostingId?: string
sourceUrl?: string
sourcePlatform?: string
```

同时保留以下岗位快照字段：

- `companyName`
- `companyIntro`
- `jobTitle`
- `jobRequirements`
- `base`
- `salaryRange`
- `sourcePlatform`
- `sourceUrl`

## 页面与交互设计

### 岗位列表页

参考当前简历仓库列表页，采用：

- 顶部搜索筛选区
- 卡片宫格列表
- 分页
- 新增岗位按钮

导航层级要求：

- 岗位列表作为与“简历管理”同级的一级导航显示
- 岗位详情页作为隐藏详情路由存在，并高亮一级“岗位列表”菜单

筛选项建议：

- 关键词：公司名称 / 岗位名称
- 来源平台：Boss / 官网 / 牛客 / 内推 / 其他
- 岗位状态：待投递 / 已投递 / 不合适 / 已关闭

卡片展示字段建议：

- 公司名称
- 岗位名称
- base
- 薪资范围
- 来源平台
- 岗位状态
- 创建时间

视觉要求：

- 岗位卡片尺寸与简历仓库卡片保持一致
- 卡片内容区使用稳定高度，避免不同文案长度导致卡片高度不一致
- 列表容器左右保留合适留白，保证第一张和最后一张卡片不贴边
- 卡片 footer 贴底，保持操作区位置一致

卡片操作：

- 查看
- 编辑
- 删除
- 查看原岗位

### 岗位创建 / 编辑

岗位创建建议使用弹窗，并提供两种入口：

1. 手动录入
2. 输入链接后解析导入

交互建议：

- 弹窗顶部提供两种模式切换或并列入口
- 当用户输入来源链接并点击“解析导入”后，调用 `POST /api/job-postings/parse`
- 解析成功后，将结构化字段回填到弹窗表单
- 用户可继续手动补充或修正字段

字段建议：

- 公司名称
- 公司介绍
- 岗位名称
- 岗位要求
- base
- 薪资范围
- 来源平台
- 来源链接
- 岗位状态
- 备注

### 岗位详情页

岗位详情页采用以下结构：

1. 顶部摘要区：公司名称、岗位名称、状态标签
2. “岗位基础信息与操作”合并区域
3. 关联投递记录列表

合并区域中需要同时展示：

- 公司介绍
- 岗位要求
- 来源链接
- 备注
- 创建投递记录按钮
- 编辑岗位按钮
- 查看原岗位按钮

交互规则：

- `创建投递记录`：跳转到投递记录创建页，并自动带入岗位字段
- `编辑岗位`：弹出岗位编辑弹窗
- `查看原岗位`：新开页打开 `sourceUrl`

### 从岗位到投递的跳转

用户从岗位详情页点击 `创建投递记录` 后：

1. 跳转到投递记录创建页
2. 自动带入 `jobPostingId`
3. 自动回填岗位相关字段
4. 用户只需补充简历版本、投递日期、当前状态、备注等信息

如果后续支持从投递记录创建页选择岗位库，也应复用同一套回填逻辑。

## API 与 Mock 设计

新增模块建议：

- `src/models/job-posting.ts`
- `src/api/job-postings.ts`
- `src/store/modules/job-posting.ts`
- `src/views/job-posting/list.vue`
- `src/views/job-posting/detail.vue`
- `src/components/JobPostingCard/index.vue`
- `mock/job-postings.ts`

接口建议：

- `GET /api/job-postings`
- `GET /api/job-postings/:id`
- `POST /api/job-postings`
- `PUT /api/job-postings/:id`
- `DELETE /api/job-postings/:id`
- `POST /api/job-postings/parse`

解析接口返回结构化岗位数据，前端无需关心解析来源细节，只按统一 JSON 回填表单。

## 风险与权衡

- [岗位信息与投递记录快照可能出现双份数据] -> 缓解方式：明确岗位库负责“当前岗位信息”，投递记录负责“历史投递快照”
- [岗位解析功能容易让人误解为前端抓取] -> 缓解方式：在设计和接口定义中明确前端只消费解析结果
- [岗位状态和投递状态容易混淆] -> 缓解方式：在模型和页面文案上明确岗位状态只描述岗位池状态，不描述面试流程
- [弹窗字段较多，可能显得拥挤] -> 缓解方式：在弹窗中按“岗位信息 / 来源信息 / 补充信息”分组

## 迁移方案

当前系统使用 Mock 数据，不涉及持久化迁移。实现时新增岗位库 mock 数据和解析接口 mock，同时补充投递记录的岗位关联字段。前端页面和路由按新增模块接入即可。

如果后续需要回滚，可移除岗位库路由、页面、Store、Mock 接口，以及投递记录中的岗位关联入口。由于岗位库是新增模块，与现有简历管理解耦较好，回滚风险较低。
