## 背景

本项目当前的简历仓库模型通过 `JobType` 区分求职类型，已有取值为：

- `campus`：校招
- `social`：社招

现有实现中，求职类型已经贯穿简历仓库的模型、Mock 数据、列表筛选、创建 / 编辑表单、卡片展示和详情页展示。新增“实习”不需要引入新的模块，也不需要改变简历仓库和简历版本之间的关系，但必须保证所有消费 `JobType` 的位置同步扩展，避免出现数据可保存但页面无法展示、可筛选但卡片文案错误等不一致问题。

## 目标 / 非目标

**目标：**

- 将“实习”作为与“校招”“社招”同级的求职类型。
- 让简历仓库创建、编辑、筛选、卡片展示和详情展示都支持“实习”。
- 保持 `JobType` 作为统一枚举来源，避免在页面里散落硬编码判断。
- 更新 Mock 数据，提供至少一条实习类型的简历仓库样例。
- 同步更新文档中“校招 / 社招”的相关描述。

**非目标：**

- 本次不新增独立的实习模块。
- 本次不调整简历仓库、简历版本、投递记录和岗位库之间的业务关系。
- 本次不为实习类型引入特殊流程、特殊字段或特殊权限。
- 本次不扩展岗位库的岗位类型字段；岗位库若要支持“校招 / 社招 / 实习”分类，需要另起 change。

## 关键设计决策

`internship` 作为 `JobType` 的第三个枚举值。
原因：现有 `campus`、`social` 都使用英文枚举值存储，新增类型应保持同一风格，避免在数据层使用中文值。
备选方案：
- 使用 `intern`：更短，但语义更偏“实习生身份”，不如 `internship` 对应“实习求职类型”清晰。
- 使用中文值 `实习`：展示直接，但与现有枚举风格不一致。

统一封装求职类型展示文案和标签类型。
原因：当前卡片和详情页通过二分判断显示“校招 / 社招”，新增第三种类型后继续写三元判断会增加遗漏风险。建议在页面或轻量工具函数中集中维护 `JobType -> label` 和 `JobType -> tagType` 映射。
备选方案：
- 在每个页面单独判断：改动局部更小，但后续再扩展类型时容易产生展示不一致。

“实习”在筛选与表单中作为普通选项处理。
原因：本次只是扩展类型枚举，不引入新流程。实习仓库与校招、社招仓库一样支持创建、编辑、删除、查看和版本管理。
备选方案：
- 给实习增加特殊字段或特殊页面：超出本次范围，也会把一个轻量类型扩展变成流程改造。

Mock 数据增加实习样例。
原因：列表筛选和卡片展示需要可见样例，便于本地验证“实习”选项可查询、可展示。
备选方案：
- 只扩展类型但不加样例：仍可通过创建验证，但本地首屏验证成本更高。

## 数据模型设计

### 求职类型枚举

建议将 `JobType` 扩展为：

```ts
export enum JobType {
  CAMPUS = 'campus',
  SOCIAL = 'social',
  INTERNSHIP = 'internship'
}
```

语义说明：

- `campus`：校招
- `social`：社招
- `internship`：实习

### 简历仓库实体

`ResumeRepo.jobType` 仍复用 `JobType`，无需新增字段：

```ts
jobType: JobType
```

### 查询参数

`ResumeRepoQueryParams.jobType` 仍复用 `JobType`，允许传入 `internship`：

```ts
jobType?: JobType
```

## 页面与交互设计

### 简历仓库列表筛选

求职类型筛选下拉应包含：

- 校招：`campus`
- 社招：`social`
- 实习：`internship`

选择“实习”后，列表只展示 `jobType === 'internship'` 的简历仓库。

### 创建 / 编辑简历仓库

求职类型单选项应包含：

- 校招
- 社招
- 实习

默认值可以继续保持“校招”，避免改变既有创建习惯。

### 简历仓库卡片

卡片中的求职类型标签需要支持：

- 校招
- 社招
- 实习

标签颜色需要与既有 Element UI 风格一致，并与校招、社招形成可区分状态。建议：

- 校招：`success`
- 社招：`primary`
- 实习：`warning`

### 简历仓库详情页

详情页“求职类型”字段需要正确展示“实习”，并使用与卡片一致的标签颜色。

## 实现方案

建议按以下顺序修改：

1. 更新 `src/models/resume-repo.ts`，新增 `JobType.INTERNSHIP`。
2. 更新 `mock/resume-repos.ts`，让 Mock 类型声明支持 `internship`，并增加实习样例数据。
3. 在简历仓库列表页新增筛选和表单选项。
4. 在卡片组件和详情页中将二分判断改为映射函数或映射对象。
5. 更新文档中“校招 / 社招”的求职类型描述。

## 风险与权衡

- [展示逻辑遗漏] -> 缓解：统一梳理所有 `JobType.CAMPUS ? 校招 : 社招` 的二分判断并改为三值映射。
- [Mock 类型与模型类型不一致] -> 缓解：Mock 中不要继续手写 `'campus' | 'social'`，应同步加入 `'internship'` 或直接复用模型类型。
- [文档口径不一致] -> 缓解：更新 PRD 和项目概要中关于简历仓库求职类型的描述。
- [后续岗位库也需要岗位类型] -> 缓解：本次明确不扩展岗位库，避免 scope 漂移；后续如需要单独提出 change。

## 迁移计划

该变更不涉及已有数据迁移。已有 `campus` 和 `social` 数据保持不变。新增 `internship` 后，新创建的简历仓库可以使用该类型；若已有用户希望将某个仓库改为实习类型，可通过编辑简历仓库完成。
