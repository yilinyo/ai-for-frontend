## 为什么要做

当前系统已经有简历仓库和简历版本创建能力，也有一个挂在“简历管理”下的个人信息页面雏形。但从用户使用路径看，个人信息是账号级基础资料，不应该作为“简历管理”的子功能存在；同时，用户在创建简历版本时还需要重复填写姓名、电话、邮箱、求职意向等基础信息，降低了简历创建效率。

本次变更需要将个人信息页调整为与“引导页 / Guide”同级的一级入口，并以表单方式维护当前账号的结构化个人资料。后续创建简历版本时，系统根据当前账号已保存的个人信息自动生成默认简历内容草稿，用户可以继续在简历编辑器中自由修改。

该功能先采用 Mock-first 实现，Mock 数据需要按当前登录账号隔离，模拟后续后端按账号独立存储个人资料的行为。

## 变更内容

- 将个人信息页作为左侧一级菜单入口展示，与“引导页 / Guide”同级。
- 移除“简历管理”下的个人信息入口，避免同一功能在两个导航层级重复出现。
- 扩展个人信息表单字段，支持维护姓名、年龄、邮箱、电话、求职意向、头像、所在地、个人优势，以及多段教育经历。
- 教育经历支持学校、学历、专业、入学时间、毕业时间；同一种学历只允许维护一段经历。
- 扩展用户模型、API 类型、Vuex 状态和 Mock 数据，使个人资料按当前登录账号读取与保存。
- 创建简历版本时，若是新建模式且内容为空，自动根据当前账号个人信息生成默认简历内容草稿。
- 自动生成后的草稿仍放在现有简历编辑器中，用户可以继续编辑修改。

## 能力定义

### 新增能力

- `user-profile-management-as-top-level-entry`：个人信息作为账号级资料，通过一级导航入口进入，并以结构化表单维护。
- `resume-version-prefill-from-user-profile`：创建简历版本时根据当前账号个人信息自动生成可编辑的默认简历内容草稿。

### 修改能力

- `resume-navigation-structure`：移除“简历管理”下的个人信息子入口，个人信息不再作为简历管理二级菜单。
- `mock-user-profile-account-isolation`：Mock 用户资料读取和更新需要基于当前登录 token 对应账号，模拟后端账号隔离。

## 影响范围

- 影响数据模型：扩展 `UserProfile` 和 `UpdateProfileRequest` 字段。
- 影响 API 与 Mock：继续使用 `GET /api/user/profile`、`PUT /api/user/profile`，但扩展请求 / 响应字段与 Mock 用户样例。
- 影响状态管理：用户 Vuex 模块继续作为个人资料缓存来源。
- 影响路由与导航：新增或调整顶层个人信息入口，并移除 `/resume/profile` 子路由入口。
- 影响页面：扩展个人信息表单字段和校验。
- 影响简历创建：创建简历版本时自动填充默认内容草稿；编辑已有简历版本时不自动覆盖。
- 当前阶段不引入真实后端，仍通过 Mock 数据完成闭环。

## 非目标

- 不把个人信息页改成 Markdown 编辑器。
- 不在本次变更中重构整个简历编辑器为结构化表单式简历编辑。
- 不实现多份个人资料模板、隐私权限配置或云端头像上传。
- 不改变简历版本的存储结构，简历内容仍沿用当前 `content` 字段。
- 不把岗位、投递、面试信息自动写入简历草稿。
