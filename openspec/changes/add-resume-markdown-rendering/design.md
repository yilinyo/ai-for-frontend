## Context

当前简历详情页通过 `content.replace(/\\n/g, '<br/>')` 将文本换行直接转成 `<br/>`，因此 Markdown 标记语法不会被解析，用户会在页面和 PDF 中直接看到 `#`、`##`、`-` 等原始标记。编辑页 `src/views/resume-version/editor.vue` 也仍然使用纯 `textarea` 输入，并通过单独弹窗做预览，既没有真实 Markdown 渲染，也不支持在编辑区里快速切换编辑/预览状态。上一轮 PDF 导出已经建立了隐藏导出容器和浏览器端导出链路，但导出的内容仍然依赖这份纯文本渲染结果，因此问题同时存在于编辑、页面预览和导出文档中。

这个变更需要同时影响三个场景：
- 编辑页中的编辑模式与预览模式切换
- 页面内的简历预览
- PDF 导出的隐藏文档容器

为了避免三个场景分别维护不同的内容逻辑，设计上应优先让它们共用同一套 Markdown 渲染与样式约束。

## Goals / Non-Goals

**Goals:**
- 将简历内容按真实 Markdown 语义渲染，而不是展示原始标记文本。
- 让简历编辑页支持显式的“编辑模式 / 预览模式”切换，交互接近 VSCode 的 Markdown 使用方式。
- 保证编辑页预览、页面预览与 PDF 导出的内容结构一致。
- 为常见 Markdown 元素提供适合简历阅读的样式表现。
- 控制渲染后的 HTML 安全边界，避免直接注入不可信内容。

**Non-Goals:**
- 引入完整的富文本编辑器重构。
- 支持所有扩展型 Markdown 特性，如流程图、数学公式、表格插件生态等。
- 改造简历编辑页为真正的所见即所得编辑器。
- 实现真正的左右双栏实时编辑器；本次以单区域模式切换为主，更接近 VSCode 的“编辑/预览视图切换”。
- 在本次变更中重做整个 PDF 模板系统。

## Decisions

引入专用 Markdown 渲染库并在前端统一解析。
原因：当前代码没有 Markdown 解析能力，继续手工处理标题、列表等语法会快速失控。使用成熟解析库可以稳定支持标题、段落、列表、强调、引用、代码块等常见语法。
备选方案：
- 手写正则转换：实现成本看似低，但很难正确覆盖嵌套列表、强调和代码块等结构。
- 继续纯文本渲染：无法满足用户要“真实 Markdown”的核心诉求。

引入 HTML 清洗步骤，避免将解析后的原始 HTML 直接注入页面。
原因：当前页面使用 `v-html`，如果未来内容里混入原始 HTML，直接渲染会带来安全风险。解析后统一清洗可以把渲染边界控制在可接受范围内。
备选方案：
- 完全信任内容来源：短期实现更快，但风险会扩散到页面预览和 PDF 导出两个入口。

页面预览和 PDF 导出共用同一个渲染结果字段。
原因：如果页面和导出各自解析 Markdown，后续很容易出现渲染配置不一致、样式不一致和问题难排查的情况。共用一个 `renderedContent` 输出更容易保证一致性。
备选方案：
- 页面与导出分别解析：灵活但重复，容易出现行为漂移。

编辑页使用显式模式切换，而不是继续保留单独弹窗预览。
原因：用户期望的是像 VSCode 一样围绕编辑区域快速切换“写作”和“阅读”两种状态，而不是触发一次性的临时弹窗。使用顶部图标按钮切换编辑模式（小铅笔）与预览模式（小书本）更符合这类 Markdown 工作流。
备选方案：
- 保留弹窗预览：实现最省事，但交互割裂，和用户期望不一致。
- 直接做左右分栏：体验也好，但范围更大，会引入额外布局和响应式复杂度。

为 Markdown 结果定义一套“简历化”样式，而不是直接套用博客风格。
原因：简历阅读强调层级清晰、密度适中和打印友好，默认 Markdown 样式往往过于松散或偏内容社区化。需要用更紧凑的标题间距、列表缩进和段落节奏来适配简历场景。
备选方案：
- 使用通用 Markdown 样式库默认主题：实现快，但视觉上不一定适合简历和 PDF 打印。

## Risks / Trade-offs

- [Markdown 解析库会增加前端包体积] -> Mitigation: 选择能力足够但相对轻量的解析库，并复用现有页面逻辑避免重复实现。
- [清洗规则过严可能过滤掉部分用户期望的内联 HTML] -> Mitigation: 本次先以安全和稳定为优先，只保证标准 Markdown 语法表现正确。
- [页面样式与 PDF 渲染引擎存在细微差异] -> Mitigation: 页面和导出共用内容结构，但分别为屏幕与打印容器补充适配样式。
- [编辑页模式切换可能让用户误以为预览内容可直接编辑] -> Mitigation: 明确区分编辑态与预览态的图标、按钮状态和只读视觉样式。
- [复杂 Markdown 内容在导出时可能分页不理想] -> Mitigation: 为标题、列表、代码块等主要结构补充 page-break 规则并用长简历样本验证。

## Migration Plan

该变更不涉及数据迁移。上线时只需增加前端依赖、更新简历详情页渲染逻辑和 PDF 导出样式。如果出现兼容性问题，可回滚到原有纯文本渲染实现，因为接口和存储内容格式均不需要变化。
